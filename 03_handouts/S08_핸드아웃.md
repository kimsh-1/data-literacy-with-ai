# S08. 좋은 분석 프롬프트 설계

## 사용할 데이터

좋은 분석 프롬프트는 AI에게 보내는 분석 요청서입니다. 수업에서는 고객 설문 응답을 가지고, AI가 주관식 문장을 어떻게 주제와 감성으로 나누는지 직접 확인해 봅니다.

기본 실습은 수업 홈의 데이터 받기 페이지에서 내려받은 고객 설문 정제 파일로 진행합니다. CSV는 엑셀로도 열리는 표 파일입니다. 엑셀 파일이 더 익숙한 분은 같은 데이터의 엑셀 파일을 받아 사용해도 됩니다.

보조 확인용으로는 데이터 상태가 일부러 조금 섞여 있는 고객 설문 파일을 봅니다. 선택 확장용으로는 영화 장르 데이터 파일을 사용할 수 있습니다. 선택 확장 파일은 시간이 남을 때만 다룹니다.

수업에서 확인한 데이터 규모입니다.

| 파일 | 행수 | 열수 | 용도 |
|---|---:|---:|---|
| `survey_responses_clean.csv` | 1,500 | 8 | 주관식 주제분류와 감성요약 |
| `survey_responses_dirty.csv` | 1,510 | 8 | 데이터 상태를 프롬프트에 알려야 하는 이유 설명 |
| `dacon_movies_train.csv` | 600 | 12 | 시간이 남을 때 장르 데이터 확장 실습 |

AI 파일 업로드 기능은 쓰는 도구와 요금제에 따라 막혀 있을 수 있습니다. 업로드가 되면 데이터 받기 페이지에서 내려받은 고객 설문 정제 파일을 올리고, 업로드가 안 되면 아래 표본 18행을 복사해서 씁니다. 분석 재료를 먼저 분명히 해야 AI 답변도 어디까지 믿을 수 있는지 판단할 수 있습니다.

## 오늘의 목표

오늘은 AI에게 설문 주관식 응답을 분석시켜 봅니다. 다만 목표는 AI 답변을 그대로 믿는 것이 아닙니다. 좋은 분석 프롬프트를 설계해서, 사람이 다시 확인할 수 있는 표 형태의 결과를 받아 내는 연습을 합니다.

좋은 분석 프롬프트에는 세 가지가 들어갑니다. 엑셀에서 피벗테이블을 만들 때 행, 열, 값을 지정하듯이 AI에게도 기준을 알려 주는 겁니다.

| 요소 | 확인 질문 |
|---|---|
| 맥락 | 이 데이터가 무엇이고 왜 분석하는가 |
| 질문 | AI가 정확히 무엇을 해야 하는가 |
| 출력형식 | 결과를 어떤 표와 문장으로 받을 것인가 |

맥락은 “이 표가 어떤 표인지” 알려 주는 설명입니다. 질문은 “무엇을 해 달라는지”를 좁히는 말입니다. 출력형식은 “결과를 어떤 모양으로 받을지” 정하는 약속입니다.

## 따라하기 1. 나쁜 프롬프트와 좋은 프롬프트 비교

### 1단계. 파일 또는 표본 준비

먼저 AI에게 줄 재료를 정합니다. 전체 파일을 올릴지, 표본 18행만 붙여 넣을지에 따라 결과의 의미가 달라지기 때문입니다.

수업 홈의 데이터 받기 페이지에서 고객 설문 정제 파일을 내려받아 준비해 보세요. 파일 업로드가 편하면 파일을 사용하고, 업로드가 막혀 있으면 위의 표본 18행을 복사하면 됩니다.

둘 중 하나를 선택합니다.

| 상황 | 진행 방법 |
|---|---|
| AI 파일 업로드 가능 | `survey_responses_clean.csv`를 업로드한다 |
| AI 파일 업로드 불가 | 위의 표본 18행을 복사한다 |

업로드를 했다면 AI에게 파일 이름과 컬럼명을 인식했는지 먼저 확인하게 합니다. 이 확인은 AI가 다른 데이터로 착각한 채 분석을 시작하지 않게 막는 안전장치입니다.

```text
첨부한 파일의 행 수와 컬럼명을 확인해줘.
그 다음 내가 지시하기 전까지 분석 결론은 내지 마.
```

### 2단계. 프롬프트 A 실행

이 단계에서는 일부러 짧은 프롬프트를 써 봅니다. 그래야 “분석해줘”만 말했을 때 결과가 얼마나 흔들리는지 직접 볼 수 있습니다.

아래처럼 짧게 요청합니다.

```text
이 설문 데이터를 분석해줘.
```

결과를 읽고 아래 질문에 답해 보세요. AI가 그럴듯하게 말했는지보다, 사람이 다시 확인할 수 있는지 보는 것이 중요합니다.

1. 어떤 기준으로 주제를 나눴는가?
2. 원문 근거가 보이는가?
3. 결과를 엑셀에 붙여 넣어 피벗테이블로 다시 집계할 수 있는가?

프롬프트 A는 대개 설명은 그럴듯합니다. 하지만 기준이 분명하지 않고, 출력형식도 일정하지 않은 경우가 많습니다. 표로 다시 쓰기 어렵다면 업무에 바로 쓰기 힘듭니다.

### 3단계. 프롬프트 B 실행

이번에는 AI에게 맥락, 질문, 출력형식을 함께 줍니다. 분석을 맡기는 것이 아니라, 분석 기준을 같이 정해 주는 단계입니다.

아래 프롬프트를 실행합니다. 업로드를 하지 않았다면 마지막에 표본 18행을 붙입니다.

```text
너는 고객경험 분석가다.
데이터는 온라인 쇼핑 서비스의 고객 설문이다. 한 행은 응답 1건이다.
주요 컬럼은 respondent_id, age_group, channel, nps, satisfaction, free_text다.

목표는 free_text를 읽고 고객 의견을 운영 개선에 쓸 수 있게 정리하는 것이다.
각 응답을 주제와 감성으로 분류해줘.

주제는 6-8개로 만들되 너무 세분화하지 마라.
감성은 긍정, 부정, 혼합, 중립 중 하나로 정하라.
NPS와 free_text의 분위기가 어긋나면 혼합 또는 검토 필요로 표시하라.

출력은 표로 작성하라.
표 컬럼은 respondent_id, topic, sentiment, evidence, confidence, needs_review로 해라.
evidence에는 판단 근거가 된 원문 일부를 넣어라.
confidence는 높음, 중간, 낮음 중 하나로 표시하라.
```

프롬프트 A와 B의 차이를 확인합니다. 특히 `evidence` 열이 있는지 보세요. `evidence`는 판단 근거가 된 원문 일부입니다. 사전에서 단어를 찾듯이, 나중에 원문과 AI 판단을 다시 대조할 수 있게 해 줍니다.

| 비교 항목 | A | B |
|---|---|---|
| 데이터 맥락이 있는가 |  |  |
| 질문이 구체적인가 |  |  |
| 표 출력이 가능한가 |  |  |
| 원문 근거가 있는가 |  |  |
| 사람이 다시 검토할 행을 찾을 수 있는가 |  |  |

## 따라하기 2. 단계적으로 심화하기

한 번에 전체 분석을 요구하지 않고 세 번에 나누어 요청합니다. 먼저 분류 기준을 만들고, 그 기준으로 행을 나누고, 마지막에 요약을 받는 순서입니다. 기준 없이 바로 요약부터 받으면 나중에 “왜 이렇게 나눴지?”를 확인하기 어렵습니다.

### 1단계. 코드북 먼저 만들기

코드북은 분류 기준표입니다. 예를 들어 “배송”, “결제”, “앱 사용성”처럼 어떤 의견을 어느 주제로 넣을지 정해 둔 작은 사전이라고 생각하면 됩니다.

```text
먼저 이 설문 free_text를 분류할 주제 코드북을 만들어줘.

조건:
- 주제는 6-8개로 제한한다.
- 각 주제마다 정의와 포함 예시를 적는다.
- 서로 겹치기 쉬운 주제는 구분 기준을 적는다.
- 아직 행별 분류는 하지 말고 코드북만 제안한다.
```

확인할 점입니다.

- 주제가 너무 많지 않은가
- "기타"가 남용되지 않도록 기준이 있는가
- 배송, 앱 사용성, 결제와 혜택, 고객지원처럼 업무 조치로 이어지는 범주인가

주제가 너무 많으면 나중에 집계가 어려워집니다. 반대로 너무 넓으면 실제로 무엇을 고쳐야 할지 흐려집니다. 그래서 6-8개 정도로 제한해 보는 겁니다.

### 2단계. 같은 코드북으로 행별 분류하기

코드북이 괜찮다면 이어서 요청합니다. 같은 기준표를 계속 써야 행마다 판단이 덜 흔들립니다.

```text
방금 만든 코드북을 사용해서 아래 데이터의 각 행을 분류해줘.

규칙:
- 한 행의 topic은 하나만 고른다.
- 감성은 긍정, 부정, 혼합, 중립 중 하나로 정한다.
- 판단이 애매하면 confidence를 낮음으로 두고 needs_review를 예로 표시한다.
- 원문에 없는 내용을 추측하지 않는다.

출력 표 컬럼:
respondent_id, topic, sentiment, evidence, confidence, needs_review
```

결과를 받은 뒤 `respondent_id`가 빠진 행이 없는지 확인합니다. `respondent_id`는 원본과 AI 결과를 이어 주는 열입니다. 이름표가 빠지면 나중에 어느 응답을 말하는지 찾기 어렵습니다.

### 3단계. 요약과 검토 포인트 요청하기

이 단계에서는 행별 분류 결과를 보고 요약을 받습니다. 먼저 재료가 전체 파일인지 표본 18행인지 구분해야 합니다. 전체와 표본은 말할 수 있는 범위가 다릅니다.

파일 업로드를 한 경우:

```text
이번에는 전체 1,500행 기준으로 요약해줘.

출력:
1. 주제별 응답수와 비중
2. 주제별 평균 NPS와 평균 satisfaction
3. 부정 또는 혼합 응답이 많은 주제 3개
4. 대표 근거문구 2개씩
5. 사람이 다시 확인해야 할 분류 기준 3개

숫자는 가능한 한 표로 보여주고, 결론은 5문장 이내로 써줘.
```

파일 업로드가 안 된 경우:

```text
위 표본 18행 기준으로만 요약해줘.
전체 1,500행의 결론처럼 말하지 말고, 반드시 "표본 18행 기준"이라고 표시해줘.
```

중요한 건 표본 분석과 전체 분석을 갈라 보는 일입니다. 표본 18행만 보고 전체 고객의 결론인 양 쓰면 안 됩니다. 작은 표본은 연습과 점검에는 좋지만, 전체 고객의 경향이라고 말하려면 전체 1,500행을 봐야 합니다.

## 따라하기 3. AI 답변 검증하기

AI 답변을 받은 뒤에는 검증을 시킵니다. 검증은 AI를 의심하려고 하는 일이 아니라, 사람이 다시 봐야 할 행을 빨리 찾기 위한 과정입니다.

아래 프롬프트로 AI가 만든 분류 결과를 스스로 다시 살펴보게 합니다.

```text
방금 만든 분류 결과에서 사람이 다시 확인해야 할 가능성이 큰 행 5개를 골라줘.

기준:
- NPS는 낮지만 free_text는 긍정적으로 보이는 행
- NPS는 높지만 free_text에 불만이나 개선 요청이 있는 행
- 하나의 문장에 긍정과 부정이 함께 있는 행
- 주제를 하나로 고르기 어려운 행

출력은 respondent_id, 검토 이유, 원문 근거로 작성해줘.
```

이번 표본에는 다음처럼 점수와 문장이 어긋나는 사례가 있습니다.

- `R2025-001293`: NPS는 2점이지만 "부담이 없었습니다"라는 긍정 표현이 있다.
- `R2025-000534`: NPS는 9점이지만 알림이 너무 잦다는 불편이 있다.
- `R2025-001185`: NPS는 8점이지만 결제 오류와 재시도가 언급된다.

이런 행을 집어내는 것이 좋은 프롬프트의 성과입니다. 점수만 보면 놓치고, 문장만 봐도 놓칠 수 있는 부분을 함께 보게 만들기 때문입니다.

## 직접 해보기. 프롬프트 3버전 A/B/C 비교

같은 파일 또는 같은 표본 18행을 사용해 프롬프트 3개를 직접 만들어 봅니다. 같은 재료를 써야 프롬프트 차이만 비교할 수 있습니다.

### 조건

1. 프롬프트 A는 일부러 짧게 만든다.
2. 프롬프트 B는 맥락, 질문, 출력형식을 넣는다.
3. 프롬프트 C는 코드북 설계, 행별 분류, 요약, 검증 요청으로 단계화한다.
4. 세 프롬프트의 결과를 비교한다.

프롬프트 A는 기준이 부족한 답이 어떻게 나오는지 보는 용도입니다. 프롬프트 B는 업무 지시서처럼 조건을 갖춘 요청입니다. 프롬프트 C는 분석을 한 번에 끝내지 않고, 기준 만들기부터 검증까지 나누어 진행하는 방식입니다.

### 제출 형식

아래 표를 채웁니다. 빈칸에는 “좋음”, “보통”, “부족”처럼 적어도 되고, 짧은 설명을 붙여도 됩니다.

| 항목 | 프롬프트 A | 프롬프트 B | 프롬프트 C |
|---|---|---|---|
| 맥락이 충분한가 |  |  |  |
| 질문이 구체적인가 |  |  |  |
| 출력형식이 표로 고정되는가 |  |  |  |
| 원문 근거가 포함되는가 |  |  |  |
| 검토 필요 행을 찾는가 |  |  |  |
| 업무에 바로 쓰기 좋은가 |  |  |  |

마지막에 아래 문장으로 적습니다.

`내가 선택한 최종 프롬프트는 __이며, 이유는 __이다. 특히 __ 열이 있어 원본과 대조할 수 있다.`

여기서 좋은 답은 “AI가 멋진 문장을 썼다”가 아닙니다. 원본과 대조할 수 있고, 표로 다시 집계할 수 있고, 사람이 검토해야 할 행을 찾을 수 있는 답입니다.

## 자주 생기는 문제

### AI가 전체 결론처럼 말하지만 나는 표본만 넣었다

프롬프트에 "표본 18행 기준이라고 명시하라"를 넣습니다. 표본 분석을 전체 결론처럼 말하면 과장입니다. 표본은 맛보기 한 숟가락이지, 냄비 전체를 다 확인한 것은 아닙니다.

### AI가 원문에 없는 내용을 추측한다

"원문에 없는 원인, 의도, 고객 특성은 추측하지 말라"를 조건에 넣습니다. 결과에 `evidence` 열을 요구하면 추측을 줄일 수 있습니다. 원문 근거가 없는 말은 나중에 확인할 길이 없기 때문입니다.

### 주제가 너무 많아진다

"주제는 6-8개로 제한한다"를 넣습니다. 분류가 너무 세밀하면 나중에 피벗테이블로 집계하기 어렵습니다. 사전을 만들 때 단어가 너무 잘게 쪼개져 있으면 찾기 힘든 것과 비슷합니다.

### 같은 문장을 볼 때마다 분류가 조금씩 달라진다

먼저 코드북을 확정하고, 그 코드북을 기준으로 다시 분류하라고 요청합니다. 중요한 업무라면 표본을 사람이 검토해 코드북을 고친 뒤 전체에 적용합니다. 기준표를 고정해야 같은 문장을 볼 때 판단도 덜 흔들립니다.

### 개인정보가 포함된 실제 설문이면 어떻게 하나

실제 업무 데이터에는 이름, 전화번호, 이메일, 주소 같은 개인정보를 AI에 넣지 않습니다. 이번 수업 데이터는 합성 데이터지만, 실무에서는 업로드 전에 민감정보를 제거해야 합니다. 민감정보를 지운 뒤에도 분석에 필요한 열이 남아 있는지 확인하고 진행해 보세요.
