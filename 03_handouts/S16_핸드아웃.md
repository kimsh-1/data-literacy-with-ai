# S16 핸드아웃: 데이터 모델링

## 실습 목표

Power BI에서는 주문 표, 고객 표, 상품 표를 한 표로 억지로 붙이지 않고 관계로 연결할 수 있습니다. 엑셀에서 `VLOOKUP`으로 값을 찾아 붙이던 일을, Power BI에서는 표와 표 사이의 연결 규칙으로 만들어 둔다고 생각하시면 됩니다.

이 실습에서는 `shop_orders`의 세 표를 Power BI에서 관계로 연결합니다. 마지막에는 날짜 차원 테이블을 만들고, 고객 차원의 지역 필드로 매출을 나누어 봅니다. 여기서 차원은 분석 기준이 되는 설명표라는 뜻입니다. 고객, 상품, 날짜처럼 주문을 분류하는 표가 차원입니다.

## 사용할 데이터

수업 홈의 데이터 받기 페이지에서 쇼핑몰 주문 데이터의 주문, 고객, 상품 파일을 내려받습니다. CSV라는 말이 보이면 엑셀로도 열리는 표 파일이라고 이해하시면 됩니다. 초보자는 같은 내용이 담긴 Excel 파일(.xlsx)을 받아도 됩니다.

Power BI Desktop 실습에서는 같은 세 표가 들어 있는 Excel 통합 문서를 사용합니다. 파일 경로를 직접 입력하지 말고, 데이터 받기 페이지에서 내려받은 쇼핑몰 주문 Excel 파일을 선택해서 여세요.

> 안내: Mac 수강생은 Power BI 대신 Looker Studio로 개념만 학습합니다. 스타스키마와 시간지능은 1:1로 대체되지 않습니다.

Power BI에서 사용할 Excel 파일에는 세 개의 시트가 있습니다. 시트는 엑셀 아래쪽 탭 하나하나라고 생각하시면 됩니다.

| 시트 | 행 수 | 역할 |
|---|---:|---|
| `orders` | 20,000 | 주문 내역, 팩트 테이블 |
| `customers` | 3,000 | 고객 정보, 차원 테이블 |
| `products` | 200 | 상품 정보, 차원 테이블 |

`orders`는 실제 주문이 한 줄씩 쌓인 표입니다. 그래서 팩트 테이블이라고 부릅니다. 팩트는 분석할 사건이나 숫자가 쌓이는 표라는 뜻입니다. `customers`와 `products`는 주문을 설명해 주는 사전 같은 표입니다. 고객번호를 사전에서 찾으면 지역과 등급을 알 수 있고, 상품번호를 사전에서 찾으면 상품명과 가격을 알 수 있습니다.

## 따라하기

### 1단계. Power BI에 세 표 불러오기

Power BI가 분석을 하려면 먼저 표를 Power BI 안으로 가져와야 합니다. 재료를 책상 위에 올려놓아야 요리를 시작할 수 있는 것과 같습니다.

1. Power BI Desktop을 엽니다.
2. 홈 탭에서 데이터 가져오기 또는 Get data를 클릭합니다.
3. Excel 통합 문서를 선택합니다.
4. 수업 홈의 데이터 받기 페이지에서 내려받은 쇼핑몰 주문 Excel 파일을 엽니다.
5. 탐색기 창에서 다음 시트를 모두 선택합니다.
   - `orders`
   - `customers`
   - `products`
6. 데이터 변환을 클릭합니다.

여기서 데이터 변환을 누르는 이유는 바로 보고서를 만들기 전에 컬럼 형식이 맞는지 확인하기 위해서입니다. 날짜가 글자로 들어가 있거나, 숫자가 텍스트로 들어가 있으면 나중에 계산이 어긋날 수 있습니다.

### 2단계. 데이터 형식 확인하기

Power Query 편집기에서 각 표의 데이터 형식을 확인합니다. 데이터 형식은 Power BI에게 "이 값은 날짜입니다", "이 값은 숫자입니다", "이 값은 이름 같은 글자입니다"라고 알려주는 표시입니다.

주민등록번호를 숫자로 더하지 않듯이, 고객번호나 주문번호는 계산 대상이 아니므로 텍스트로 두는 편이 안전합니다. 반대로 수량과 가격처럼 계산에 쓰는 값은 숫자 형식이어야 합니다.

`orders`:

| 컬럼 | 권장 형식 |
|---|---|
| `order_id` | 텍스트 |
| `customer_id` | 텍스트 |
| `product_id` | 텍스트 |
| `order_date` | 날짜 |
| `qty` | 정수 |
| `discount_rate` | 소수 |

`customers`:

| 컬럼 | 권장 형식 |
|---|---|
| `customer_id` | 텍스트 |
| `name` | 텍스트 |
| `gender` | 텍스트 |
| `age` | 정수 |
| `region` | 텍스트 |
| `join_date` | 날짜 |
| `grade` | 텍스트 |

`products`:

| 컬럼 | 권장 형식 |
|---|---|
| `product_id` | 텍스트 |
| `product_name` | 텍스트 |
| `category` | 텍스트 |
| `price` | 정수 |
| `cost` | 정수 |

확인이 끝나면 닫기 및 적용을 클릭합니다. 이렇게 해야 Power Query에서 확인한 내용이 Power BI 모델에 반영됩니다.

### 3단계. 모델 보기에서 표 역할 확인하기

왼쪽의 모델 보기를 클릭합니다. 모델 보기는 표와 표가 어떻게 연결되어 있는지 한눈에 보는 화면입니다. 엑셀에서는 표가 시트별로 따로 보이지만, Power BI에서는 이 표들이 서로 어떤 관계인지 모델 보기에서 확인합니다.

오늘 만들 모델은 이 구조입니다.

```text
customers      1 -> *  orders  * <- 1  products
                             *
                             ^
                             1
                       날짜테이블
```

읽는 방법은 이렇게 보시면 됩니다.

- `orders`: 주문이 한 줄씩 쌓인 팩트 테이블
- `customers`: 고객번호별 지역과 등급을 가진 고객 차원
- `products`: 상품번호별 카테고리와 단가를 가진 상품 차원
- `날짜테이블`: 주문일을 연월로 묶기 위한 날짜 차원

여기서 `1`은 사전에 한 번만 나오는 값입니다. 고객 한 명은 고객 표에 한 줄만 있어야 합니다. `*`는 여러 번 반복될 수 있다는 뜻입니다. 같은 고객이 여러 번 주문하면 `orders`에는 같은 `customer_id`가 여러 줄 나올 수 있습니다.

### 4단계. 고객-주문 관계 만들기

고객 표와 주문 표를 고객번호로 연결합니다. 이 관계가 있어야 Power BI가 "이 주문은 어느 지역 고객의 주문인가요?"라는 질문에 답할 수 있습니다.

자동으로 관계가 만들어져 있으면 설정만 확인합니다. 없으면 직접 만듭니다.

1. 모델 보기에서 `customers[customer_id]`를 `orders[customer_id]`로 끌어 놓습니다.
2. 관계 설정 창에서 다음을 확인합니다.

| 항목 | 값 |
|---|---|
| 카디널리티 | 일대다 또는 1:* |
| 1쪽 | `customers` |
| 다쪽 | `orders` |
| 교차 필터 방향 | 단일 |
| 이 관계를 활성화 | 체크 |

확인을 클릭합니다.

카디널리티는 두 표의 값이 어떤 반복 구조를 가지는지 말합니다. 고객 표에는 고객번호가 한 번만 나오고, 주문 표에는 같은 고객번호가 여러 번 나올 수 있으므로 일대다 또는 1:* 관계가 맞습니다. 교차 필터 방향은 단일로 둡니다. 입문 단계에서는 고객 표가 주문 표를 걸러 주는 방향으로 생각하면 충분합니다.

### 5단계. 상품-주문 관계 만들기

이번에는 상품 표와 주문 표를 상품번호로 연결합니다. 이 관계가 있어야 주문 표에서 상품 단가를 직접 들고 있지 않아도, 상품 표의 가격을 찾아 매출을 계산할 수 있습니다.

1. `products[product_id]`를 `orders[product_id]`로 끌어 놓습니다.
2. 관계 설정 창에서 다음을 확인합니다.

| 항목 | 값 |
|---|---|
| 카디널리티 | 일대다 또는 1:* |
| 1쪽 | `products` |
| 다쪽 | `orders` |
| 교차 필터 방향 | 단일 |
| 이 관계를 활성화 | 체크 |

확인을 클릭합니다.

상품 하나는 상품 표에 한 줄만 있지만, 주문 표에서는 여러 번 팔릴 수 있습니다. 그래서 상품 표가 1쪽, 주문 표가 다쪽입니다. 사전에서 단어 하나를 여러 문장이 참조할 수 있는 것과 비슷합니다.

### 6단계. 날짜 차원 테이블 만들기

날짜 차원 테이블은 시간 분석을 위한 기준표입니다. 주문 표에도 `order_date`가 있지만, 그 컬럼은 주문이 발생한 날짜만 담고 있습니다. 월별 추세나 전월대비 같은 계산을 안정적으로 하려면 분석 기간의 날짜가 하루도 빠지지 않고 이어진 별도 날짜표가 필요합니다.

리본 메뉴에서 모델링 또는 테이블 도구의 새 테이블을 클릭합니다. 수식 입력줄에 아래 DAX를 그대로 입력합니다. DAX는 Power BI에서 측정값과 계산 테이블을 만들 때 쓰는 수식 언어입니다.

```DAX
날짜테이블 =
ADDCOLUMNS(
    CALENDAR(MIN(orders[order_date]), MAX(orders[order_date])),
    "연도", YEAR([Date]),
    "월번호", MONTH([Date]),
    "연월", FORMAT([Date], "yyyy-MM"),
    "연월정렬", YEAR([Date]) * 100 + MONTH([Date])
)
```

확인할 값:

- `날짜테이블` 행 수: 366
- 첫 날짜: 2024-01-01
- 마지막 날짜: 2024-12-31

2024년은 윤년이라서 2월 29일이 있습니다. 그래서 2024년 1월 1일부터 2024년 12월 31일까지 하루씩 만들면 366행이 됩니다. 행 수가 366이 아니면 날짜 범위가 다르게 만들어졌을 가능성이 있습니다.

### 7단계. 날짜 테이블로 지정하기

날짜 테이블을 만들기만 해서는 Power BI가 이 표를 시간 계산의 기준표로 확실히 인식하지 못할 수 있습니다. 그래서 "이 표가 날짜 기준표입니다"라고 지정해 줍니다.

1. 오른쪽 필드 창에서 `날짜테이블`을 선택합니다.
2. 테이블 도구에서 날짜 테이블로 표시 또는 Mark as date table을 클릭합니다.
3. 날짜 컬럼으로 `날짜테이블[Date]`를 선택합니다.
4. 확인을 클릭합니다.

날짜 테이블은 날짜가 빠짐없이 이어져 있어야 하고, 날짜 컬럼에 중복과 결측이 없어야 합니다. 결측은 비어 있는 값이라는 뜻입니다. 달력에서 하루가 빠지면 월별 계산이 흔들리는 것처럼, 날짜 차원도 중간에 빈 날짜가 있으면 시간 계산이 불안정해집니다.

이 실습의 `날짜테이블`은 2024년 모든 날짜를 하루 단위로 가지고 있으므로 조건을 만족합니다.

### 8단계. 날짜-주문 관계 만들기

날짜 테이블과 주문 표를 연결합니다. 이 관계가 있어야 `날짜테이블[연월]`로 2024-03을 선택했을 때, 주문 표에서 2024년 3월 주문만 걸러낼 수 있습니다.

모델 보기에서 `날짜테이블[Date]`를 `orders[order_date]`로 끌어 놓습니다.

| 항목 | 값 |
|---|---|
| 카디널리티 | 일대다 또는 1:* |
| 1쪽 | `날짜테이블` |
| 다쪽 | `orders` |
| 교차 필터 방향 | 단일 |
| 이 관계를 활성화 | 체크 |

확인을 클릭합니다.

날짜 테이블에는 하루가 한 줄씩만 있습니다. 주문 표에는 같은 날짜에 여러 주문이 있을 수 있습니다. 그래서 날짜 테이블이 1쪽, 주문 표가 다쪽입니다.

### 9단계. 정렬 열 지정하기

연월은 글자로 되어 있어서 정렬 기준을 따로 지정해 두는 편이 안전합니다. 예를 들어 사람 눈에는 2024-01, 2024-02, 2024-03 순서가 자연스럽지만, Power BI가 글자 정렬로 처리하면 의도와 다르게 보일 때가 있습니다.

1. 데이터 보기에서 `날짜테이블`을 엽니다.
2. `연월` 컬럼을 선택합니다.
3. 열 도구에서 열 기준 정렬을 클릭하고 `연월정렬`을 선택합니다.

`연월정렬`은 202401, 202402처럼 숫자로 된 정렬용 컬럼입니다. 이름표는 `연월`을 보여 주고, 실제 줄 세우기는 `연월정렬`로 하게 만드는 단계입니다. 사람에게 보이는 이름과 컴퓨터가 정렬할 기준을 따로 둔다고 생각하시면 됩니다.

### 10단계. 매출 측정값 만들기

이제 주문 수량과 상품 가격을 이용해 매출을 계산합니다. `orders` 테이블에는 매출 컬럼이 따로 없습니다. 대신 주문 한 줄마다 수량, 상품 단가, 할인율을 이용해 매출을 계산한 뒤 모두 더합니다.

`orders` 테이블을 선택하고 새 측정값을 만듭니다. 아래 DAX를 그대로 입력합니다.

```DAX
Total Sales =
SUMX(
    orders,
    orders[qty] * RELATED(products[price]) * (1 - orders[discount_rate])
)
```

이 측정값은 주문 한 줄마다 다음 계산을 한 뒤 모두 더합니다.

```text
상품 단가 × 수량 × (1 - 할인율)
```

`RELATED(products[price])`는 관계를 따라 상품 표에서 가격을 찾아오는 부분입니다. 엑셀의 VLOOKUP처럼 가격을 찾아오지만, 값을 주문 표에 복사해 붙이는 것이 아니라 관계를 따라 참조합니다.

검산용 측정값도 만듭니다. 검산은 계산 결과가 맞는지 확인하는 작은 기준을 만드는 일입니다. 체중계를 쓸 때 0점이 맞는지 먼저 보는 것과 비슷합니다.

```DAX
Order Count =
DISTINCTCOUNT(orders[order_id])
```

```DAX
Avg Order Value =
DIVIDE([Total Sales], [Order Count])
```

`Order Count`는 주문번호가 몇 개 있는지 셉니다. `Avg Order Value`는 전체 매출을 주문 수로 나누어 주문 한 건당 평균 매출을 계산합니다.

### 11단계. 카드로 전체 값 검산하기

보고서 보기로 이동합니다. 관계와 측정값이 제대로 만들어졌는지 먼저 전체 숫자로 확인합니다. 전체 숫자가 맞아야 지역별, 월별로 나누어 볼 때도 믿을 수 있습니다.

1. 카드 시각화를 추가합니다.
2. 값에 `[Total Sales]`를 넣습니다.
3. 다른 카드에 `[Order Count]`를 넣습니다.

검산 기준:

| 항목 | 값 |
|---|---:|
| `Order Count` | 20,000 |
| `Total Sales` | 1,938,489,803 |

값이 다르면 관계가 빠졌거나 `Total Sales` 수식의 `RELATED(products[price])` 부분이 제대로 작동하지 않은 것입니다. 특히 상품-주문 관계가 없으면 Power BI가 상품 가격을 주문 한 줄마다 찾아오지 못합니다.

### 12단계. 지역별 매출 표 만들기

이번에는 고객 표의 지역으로 매출을 나누어 봅니다. 여기서 중요한 점은 `orders` 표 안에 `region` 컬럼이 없다는 것입니다. 지역은 `customers` 표에 있고, 매출 계산은 `orders`와 `products`를 이용합니다. Power BI는 관계를 따라 이 세 표를 함께 사용합니다.

1. 테이블 시각화를 추가합니다.
2. 필드를 다음 순서로 넣습니다.

| 위치 | 필드 |
|---|---|
| 첫 번째 열 | `customers[region]` |
| 두 번째 열 | `[Order Count]` |
| 세 번째 열 | `[Total Sales]` |
| 네 번째 열 | `[Avg Order Value]` |

3. `[Total Sales]` 기준 내림차순으로 정렬합니다.

검산 기준:

| region | Order Count | Total Sales |
|---|---:|---:|
| 서울특별시 | 5,711 | 570,482,124 |
| 경기도 | 5,442 | 530,726,511 |
| 부산광역시 | 1,268 | 132,704,477 |

여기서 눈여겨볼 대목은 `orders`에 `region` 컬럼이 없다는 사실입니다. 지역은 `customers`에서 가져오지만, 매출은 관계를 따라 `orders` 기준으로 계산됩니다. 고객 표가 지역이라는 분류 기준을 제공하고, 주문 표가 실제 주문 건수를 제공하고, 상품 표가 가격을 제공하는 구조입니다.

## 직접 해보기

### 미션. 지역 차원으로 매출 분해하기

아래 조건을 만족하는 보고서 페이지를 만듭니다. 손으로 직접 만들어 보면서 관계가 보고서에서 어떻게 작동하는지 확인해 보세요.

1. 막대 차트를 추가합니다.
2. 축에는 `customers[region]`을 넣습니다.
3. 값에는 `[Total Sales]`를 넣습니다.
4. 도구 설명에는 `[Order Count]`, `[Avg Order Value]`를 넣습니다.
5. 슬라이서를 추가하고 `날짜테이블[연월]`을 넣습니다.
6. 선택 사항으로 범례에 `products[category]`를 넣어 지역별 매출을 상품 카테고리로 더 나눕니다.

막대 차트는 지역별 매출 차이를 눈으로 비교하기 좋습니다. 슬라이서는 원하는 월만 골라 보는 필터입니다. 예를 들어 2024-05를 선택하면 2024년 5월 주문만 기준으로 지역별 매출이 다시 계산되어야 합니다.

제출할 내용:

- 전체 매출
- 매출 1위 지역과 금액
- 매출 2위 지역과 금액
- 지역이 몇 개로 나뉘었는지
- `날짜테이블[연월]` 슬라이서가 지역별 매출을 바꾸는지 여부

통과 기준:

- 전체 매출이 1,938,489,803으로 나옵니다.
- 지역은 17개로 나옵니다.
- 매출 1위는 서울특별시, 2위는 경기도로 나옵니다.
- 슬라이서의 월을 바꾸면 지역별 매출 값이 함께 바뀝니다.
