# S08. 좋은 분석 프롬프트 설계 강사대본

## 수업 전 확인

- 기본 실습 파일: 수업 홈의 데이터 받기 페이지에서 내려받은 정리된 고객 설문 파일. CSV는 엑셀로도 열리는 표 파일입니다. 초보자는 같은 내용의 엑셀 파일을 받아도 됩니다.
- 보조 확인 파일: 수업 홈의 데이터 받기 페이지에서 내려받은 오류가 섞인 고객 설문 파일
- 선택 확장 파일: 수업 홈의 데이터 받기 페이지에서 내려받은 영화 박스오피스 학습 파일
- 정리된 설문 데이터 규모: 1,500행 × 8열
- 오류 포함 설문 데이터 규모: 1,510행 × 8열
- 선택 영화 데이터 규모: 600행 × 12열
- 정리된 설문 주요 컬럼: `respondent_id`, `age_group`, `gender`, `nps`, `satisfaction`, `channel`, `free_text`, `submitted_at`
- 오류 포함 설문 확인 사항: 중복 응답자 ID 10건, NPS 결측 60건, 주관식 결측 22건, 만족도 문자 혼입 75건, 20대 표기 혼재 90건
- 강사용 검증 코드: 정답 코드 파일을 사용합니다. 학생에게 파일 위치를 말하지 말고, 결과가 원본과 맞는지 대조하는 과정만 보여주세요.

## 0-5분. 도입

강사 멘트:

"분석을 AI에게 맡길 때 제일 먼저 봐야 할 것은 버튼이 아니라 요청 문장입니다. 지난 차시까지는 데이터를 표로 정리하고, 질문에 맞춰 요약하는 연습을 했습니다. 이제는 그 분석을 AI에게 부탁할 때 어떤 말로 부탁해야 하는지 봅니다."

"많은 분이 처음에는 '이 데이터 분석해줘'라고 씁니다. 그런데 이 말은 엑셀에서 피벗테이블을 만들면서 행, 열, 값 영역을 비워 둔 것과 비슷합니다. 빈 피벗테이블에서 좋은 결과가 나오기 어렵죠. AI도 똑같습니다. 뭔가 답은 해 주지만, 우리가 원하는 기준으로 답했다고 믿기는 어렵습니다."

시연 포인트:

- 수업 홈의 데이터 받기 페이지에서 내려받은 정리된 고객 설문 파일을 보여주고 한 행이 설문 응답 1건임을 설명한다.
- `free_text` 컬럼에 짧은 주관식 문장이 들어 있음을 보여준다.
- `nps`와 `satisfaction`은 숫자이고, `free_text`는 텍스트라 분석 방식이 다르다고 짚는다.

## 5-12분. 개념 설명: 맥락, 질문, 출력형식

강사 멘트:

"좋은 분석 프롬프트에는 세 가지가 들어갑니다. 맥락은 이 데이터가 무엇인지, 왜 분석하는지 알려 주는 부분입니다. 질문은 AI가 무엇을 해야 하는지 좁혀 주는 부분입니다. 출력형식은 표로 받을지, 요약문으로 받을지, 어떤 열을 넣을지 정하는 부분입니다."

"엑셀의 피벗테이블과 비교해 볼까요? 피벗테이블에서 행, 열, 값을 정하지 않으면 표가 만들어지지 않습니다. AI도 마찬가지입니다. 맥락, 질문, 출력형식이 없으면 분석 기준이 흔들립니다. 그래서 프롬프트는 길게 쓰는 글이 아니라, 분석 기준을 적어 주는 요청서라고 보면 됩니다."

판서 또는 화면 강조:

| 프롬프트 요소 | 설문 분석 예시 |
|---|---|
| 맥락 | 온라인 쇼핑 고객 설문, 운영 개선 목적 |
| 질문 | 주관식 응답을 주제와 감성으로 분류 |
| 출력형식 | 응답자 ID, 주제, 감성, 근거, 신뢰도, 검토 필요 |

주의해서 말할 점:

- 프롬프트는 긴 문장이 아니라 명확한 요청서다.
- AI 답변은 최종 정답이 아니라 검토 가능한 초안이다.
- 원문 근거가 없는 요약은 보고서에 바로 쓰지 않는다.

## 12-24분. 따라하기 1: 나쁜 프롬프트와 좋은 프롬프트 비교

시연 절차:

1. AI 도구를 열고 파일 업로드가 가능하면 수업 홈의 데이터 받기 페이지에서 내려받은 정리된 고객 설문 파일을 업로드한다.
2. 업로드가 불가능하면 핸드아웃의 표본 18행을 복사한다.
3. 먼저 프롬프트 A를 실행한다.

프롬프트 A:

```text
이 설문 데이터를 분석해줘.
```

강사 멘트:

"이 답변은 읽기에는 자연스러울 수 있습니다. 그런데 여기서 뭐가 빠져 있을까요? 어떤 기준으로 주제를 나눴는지, 어떤 원문을 근거로 삼았는지, 결과를 엑셀에 붙여 넣어 다시 집계할 수 있는지가 분명하지 않습니다. 보고서에 쓰려면 예쁜 문장보다 확인 가능한 표가 더 중요합니다."

이어서 프롬프트 B를 실행한다.

```text
너는 고객경험 분석가다.
데이터는 온라인 쇼핑 서비스의 고객 설문이다. 한 행은 응답 1건이다.
주요 컬럼은 respondent_id, age_group, channel, nps, satisfaction, free_text다.

목표는 free_text를 읽고 고객 의견을 운영 개선에 쓸 수 있게 정리하는 것이다.
각 응답을 주제와 감성으로 분류해줘.

주제는 6-8개로 만들되 너무 세분화하지 마라.
감성은 긍정, 부정, 혼합, 중립 중 하나로 정하라.
NPS와 free_text의 분위기가 어긋나면 혼합 또는 검토 필요로 표시하라.

출력은 표로 작성하라.
표 컬럼은 respondent_id, topic, sentiment, evidence, confidence, needs_review로 해라.
evidence에는 판단 근거가 된 원문 일부를 넣어라.
confidence는 높음, 중간, 낮음 중 하나로 표시하라.
```

시연 포인트:

- 프롬프트 B의 결과에 `respondent_id`가 있는지 확인한다.
- `evidence` 열이 원문 대조를 가능하게 한다고 설명한다.
- `needs_review`는 AI가 애매한 행을 숨기지 않게 만드는 장치라고 설명한다.

확인 질문:

"프롬프트 A와 B 중 어떤 결과가 엑셀에 다시 붙여 넣기 쉬운가요?"

기대 답:

"프롬프트 B입니다. 표 형식과 열 이름이 정해져 있기 때문입니다."

## 24-34분. 따라하기 2: 단계적으로 깊게 요청하기

강사 멘트:

"좋은 분석은 한 번의 프롬프트로 끝나지 않습니다. 특히 주관식 텍스트는 사람이 읽는 말이기 때문에 해석 기준이 먼저 필요합니다. 먼저 분류 기준을 만들고, 그 기준으로 행을 분류하고, 마지막에 요약하는 순서가 좋습니다. 키를 잴 때도 먼저 자를 맞춰 놓고 재듯이, 텍스트 분석도 먼저 기준표를 맞춰야 합니다."

시연 1: 코드북 요청

```text
먼저 이 설문 free_text를 분류할 주제 코드북을 만들어줘.

조건:
- 주제는 6-8개로 제한한다.
- 각 주제마다 정의와 포함 예시를 적는다.
- 서로 겹치기 쉬운 주제는 구분 기준을 적는다.
- 아직 행별 분류는 하지 말고 코드북만 제안한다.
```

강사 멘트:

"코드북은 분류 기준표입니다. 예를 들어 사전에서 단어 뜻을 확인하듯이, 어떤 응답을 어떤 주제로 넣을지 미리 정해 두는 표입니다. 사람이 먼저 기준을 보고 납득할 수 있어야 전체 분류를 맡길 수 있습니다. 코드북이 엉성하면 결과도 엉성해집니다."

시연 2: 행별 분류 요청

```text
방금 만든 코드북을 사용해서 아래 데이터의 각 행을 분류해줘.

규칙:
- 한 행의 topic은 하나만 고른다.
- 감성은 긍정, 부정, 혼합, 중립 중 하나로 정한다.
- 판단이 애매하면 confidence를 낮음으로 두고 needs_review를 예로 표시한다.
- 원문에 없는 내용을 추측하지 않는다.

출력 표 컬럼:
respondent_id, topic, sentiment, evidence, confidence, needs_review
```

시연 3: 검토 필요 행 요청

```text
방금 만든 분류 결과에서 사람이 다시 확인해야 할 가능성이 큰 행 5개를 골라줘.

기준:
- NPS는 낮지만 free_text는 긍정적으로 보이는 행
- NPS는 높지만 free_text에 불만이나 개선 요청이 있는 행
- 하나의 문장에 긍정과 부정이 함께 있는 행
- 주제를 하나로 고르기 어려운 행

출력은 respondent_id, 검토 이유, 원문 근거로 작성해줘.
```

시연 포인트:

- `R2025-001293`은 NPS 2점이지만 문장은 긍정 표현이 있어 검토 대상이 될 수 있다.
- `R2025-000534`는 NPS 9점이지만 알림 빈도 불만이 있어 검토 대상이 될 수 있다.
- 점수와 문장이 항상 같은 방향으로 움직이지 않는다고 설명한다.

## 34-45분. 직접 해보기: 프롬프트 3버전 비교

미션 안내 멘트:

"이제 여러분이 직접 프롬프트를 세 개 만들어 비교합니다. A는 일부러 짧게 써 보세요. B는 맥락과 질문과 출력형식을 담아 보세요. C는 여러 단계로 쪼개서 만들어 봅니다. 여기서 볼 것은 어느 답변이 더 그럴싸해 보이는지가 아닙니다. 어느 답변이 원본과 대조되고, 엑셀에 다시 붙여 넣을 수 있고, 실제 업무에 다시 쓸 수 있는지를 봅니다."

수강생 작업:

1. 같은 정리된 고객 설문 파일 또는 같은 표본 18행을 사용한다.
2. 프롬프트 A, B, C를 각각 실행한다.
3. 결과를 핸드아웃의 비교표에 기록한다.
4. 최종으로 선택할 프롬프트와 이유를 한 문장으로 쓴다.

강사가 돌아다니며 볼 점:

- 프롬프트에 데이터 맥락이 있는가
- 주제와 감성의 정의가 있는가
- 결과 표에 `respondent_id`와 `evidence`가 있는가
- 표본 분석을 전체 결론처럼 쓰지 않았는가
- 검토 필요 행을 찾도록 요청했는가

## 45-50분. 수업 마무리

강사 멘트:

"분석 프롬프트는 AI에게 보내는 분석 요청서입니다. 맥락과 질문과 출력형식이 갖춰져야 결과가 흔들리지 않습니다. 큰 질문은 한꺼번에 던지지 말고 코드북, 행별 분류, 요약, 검증으로 쪼개서 요청해 보세요."

"AI가 주관식 응답을 빠르게 읽어 주기는 합니다. 하지만 최종 분석에서는 결국 원본과 맞대 봐야 합니다. 그래서 `respondent_id`, `evidence`, `needs_review` 같은 열을 늘 챙겨서 요구하는 습관이 중요합니다. 응답자 ID는 원본 행을 찾는 주소이고, 근거 문구는 AI 판단을 확인하는 표시라고 보면 됩니다."

완료 판단 체크포인트:

수강생이 같은 데이터로 프롬프트 A/B/C를 비교하고, `respondent_id`와 `evidence`가 포함된 주제·감성 분류표를 만든 뒤 최종 프롬프트 선택 이유를 말하면 이번 차시를 완료한 것으로 본다.

## 예상 질문과 답변

### Q1. AI가 분류한 주제는 정답인가요?

A. 정답이라기보다 초안입니다. 주관식 텍스트는 해석이 들어가기 때문에 코드북을 먼저 확인해야 합니다. 그다음 표본을 사람이 검토하고, 기준이 괜찮다고 판단되면 전체에 적용하는 절차가 필요합니다.

### Q2. 감성은 NPS만 보고 정하면 안 되나요?

A. NPS는 중요한 참고값이지만 주관식 문장과 다를 수 있습니다. 낮은 NPS인데 긍정 문장을 쓴 사람도 있고, 높은 NPS인데 개선 요청을 쓴 사람도 있습니다. 그래서 감성은 점수만 보지 말고 문장 근거와 함께 봐야 합니다.

### Q3. 주제는 몇 개가 적당한가요?

A. 이번 수업에서는 6-8개가 적당합니다. 너무 적으면 여러 의견이 한 덩어리로 뭉뚱그려지고, 너무 많으면 나중에 집계하기 어렵습니다. 실무에서는 표본을 보며 주제 수를 조정합니다.

### Q4. 파일 업로드가 안 되면 어떻게 하나요?

A. 핸드아웃의 표본 18행을 복사해 실습합니다. 이때는 반드시 "표본 18행 기준"이라고 쓰게 해야 합니다. 표본 결과를 전체 1,500행의 결론으로 말하면 안 됩니다.

### Q5. AI가 원문에 없는 원인을 만들어 냅니다.

A. 프롬프트에 "원문에 없는 원인, 의도, 고객 특성은 추측하지 말라"를 넣어 보세요. 또 `evidence` 열을 요구해야 합니다. 근거 문구가 있으면 원문에 없는 판단을 더 쉽게 찾아낼 수 있습니다.

### Q6. 같은 프롬프트를 다시 실행하면 조금 다른 결과가 나옵니다.

A. 그럴 수 있습니다. 그래서 코드북을 먼저 고정하고, 출력 컬럼과 기준을 명확히 해야 합니다. 중요한 업무라면 AI 결과를 그대로 쓰지 말고 사람이 표본을 검수해야 합니다.

### Q7. 오류가 섞인 보조 설문 파일은 왜 오늘 본 실습에 쓰지 않나요?

A. 오늘 목표는 프롬프트 구조를 배우는 것입니다. 오류가 섞인 보조 설문 파일에는 결측, 중복, 문자 혼입, 표기 혼재가 있어 별도의 정제 수업이 필요합니다. 다만 데이터 상태를 프롬프트에 알려야 하는 이유를 설명할 때 보조 사례로 사용합니다.

## 강의 운영 메모

- 수강생이 AI 답변 문장만 보고 만족하면 `respondent_id`와 `evidence`가 있는지 되묻는다.
- 표본 실습자는 전체 결론처럼 쓰지 않도록 계속 확인한다.
- "좋은 프롬프트"를 길이로 판단하지 않게 한다. 길어도 기준이 흐리면 좋은 프롬프트가 아니다.
