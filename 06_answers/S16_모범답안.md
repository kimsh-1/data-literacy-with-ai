# S16 모범답안: 지역 차원으로 매출 분해

## 미션에서 확인할 것

Power BI에서 `customers`, `products`, `날짜테이블`이라는 설명용 표를 `orders` 주문 표에 연결합니다. 여기서 `orders`는 주문 한 건 한 건이 쌓여 있는 팩트 테이블입니다. 팩트 테이블은 매출처럼 계산의 바탕이 되는 사건 표라고 보면 됩니다.

수업 홈의 데이터 받기 페이지에서 내려받은 쇼핑몰 주문 파일을 사용하면 됩니다. Power BI 실습에서는 같은 데이터가 들어 있는 엑셀 파일을 받아 열어도 괜찮습니다.

여기서 가장 조심할 점은 `orders`에 지역 컬럼을 새로 붙이지 않는 것입니다. 지역은 고객 정보 표인 `customers[region]`에 이미 있습니다. 매출은 주문 표인 `orders`의 주문 행을 기준으로 계산하고, 지역별로 나누는 일은 관계가 해 줍니다. 엑셀에서 사전처럼 값을 찾아 붙이던 일을 Power BI 모델이 대신 해 주는 구조입니다.

## 정답 모델

모델 보기에서 관계가 아래 표처럼 잡혀 있어야 합니다.

| 1쪽 테이블 | 1쪽 컬럼 | 다쪽 테이블 | 다쪽 컬럼 | 상태 |
|---|---|---|---|---|
| `customers` | `customer_id` | `orders` | `customer_id` | 활성 |
| `products` | `product_id` | `orders` | `product_id` | 활성 |
| `날짜테이블` | `Date` | `orders` | `order_date` | 활성 |

세 관계 모두 카디널리티는 1:다입니다. 카디널리티는 한쪽 표의 한 줄이 다른 쪽 표의 몇 줄과 이어지는지를 말합니다. 고객 한 명은 여러 번 주문할 수 있고, 상품 하나도 여러 주문에 등장할 수 있으므로 `customers`와 `products`가 1쪽, `orders`가 다쪽입니다.

교차 필터 방향은 단일 방향으로 둡니다. 입문 단계에서는 고객, 상품, 날짜 같은 차원 표가 주문 표를 필터링한다고 생각하면 됩니다. 즉 지역을 고르면 그 지역 고객의 주문만 남고, 상품을 고르면 그 상품 주문만 남는 식입니다.

## 날짜 차원 DAX

날짜 차원은 시간 계산의 기준표입니다. `orders`에는 주문이 있었던 날짜만 들어 있지만, 날짜 차원은 분석 기간의 날짜를 하루 단위로 빠짐없이 갖고 있습니다. 달력처럼 기준을 만들어 두는 셈입니다.

새 테이블에서 아래 DAX를 그대로 사용합니다.

```DAX
날짜테이블 =
ADDCOLUMNS(
    CALENDAR(MIN(orders[order_date]), MAX(orders[order_date])),
    "연도", YEAR([Date]),
    "월번호", MONTH([Date]),
    "연월", FORMAT([Date], "yyyy-MM"),
    "연월정렬", YEAR([Date]) * 100 + MONTH([Date])
)
```

`CALENDAR`는 시작 날짜부터 끝 날짜까지 날짜를 한 줄씩 만들어 줍니다. 여기서는 주문일의 가장 이른 날짜와 가장 늦은 날짜를 기준으로 날짜표를 만듭니다. 그 뒤 `ADDCOLUMNS`로 연도, 월번호, 연월, 연월정렬을 덧붙입니다.

확인값:

| 항목 | 값 |
|---|---:|
| 첫 날짜 | 2024-01-01 |
| 마지막 날짜 | 2024-12-31 |
| 날짜 행 수 | 366 |

2024년은 윤년이라서 날짜가 366일입니다. 그래서 날짜 행 수가 366으로 나와야 맞습니다.

`날짜테이블[Date]`는 날짜 테이블로 지정합니다. 이렇게 해야 Power BI가 이 표를 시간 계산의 기준표로 인식합니다. `날짜테이블[연월]`은 `날짜테이블[연월정렬]`로 정렬합니다. `2024-10`이 `2024-02`보다 앞에 오는 식의 글자 정렬 문제를 막기 위해서입니다.

## 측정값 DAX

매출은 주문 표에 이미 들어 있는 값이 아닙니다. 주문 수량, 할인율, 상품 단가를 이용해서 계산해야 합니다. 단가는 `products` 표에 있으므로, 관계를 따라 `products[price]`를 가져옵니다.

```DAX
Total Sales =
SUMX(
    orders,
    orders[qty] * RELATED(products[price]) * (1 - orders[discount_rate])
)
```

`SUMX`는 `orders`를 한 줄씩 내려가며 계산한 뒤 모두 더합니다. 키를 재듯이 주문 한 줄마다 매출을 계산하고, 마지막에 전부 합치는 방식입니다. `RELATED(products[price])`는 현재 주문 행의 상품번호를 보고 `products` 표에서 단가를 찾아오는 역할을 합니다.

주문 건수는 주문번호의 고유 개수를 세면 됩니다.

```DAX
Order Count =
DISTINCTCOUNT(orders[order_id])
```

평균 주문 금액은 전체 매출을 주문 건수로 나눕니다. `DIVIDE`를 쓰면 나눗셈에서 문제가 생겼을 때 더 안전하게 처리할 수 있습니다.

```DAX
Avg Order Value =
DIVIDE([Total Sales], [Order Count])
```

이 표에서 중요한 점은 `orders` 안에 `region` 컬럼이 없다는 사실입니다. 지역은 `customers`에서 가져왔고, 매출은 `orders`를 기준으로 계산했습니다. 두 표가 `customer_id`로 잘 연결되어 있기 때문에 이런 분석이 가능합니다.

| 매출 1위 지역 | 서울특별시 |

전체 주문 건수는 20,000이어야 합니다. 전체 매출은 1,938,489,803이어야 합니다. 지역 수는 17개로 나와야 합니다. 이 값이 다르면 관계가 빠졌거나, 매출 측정값에서 단가를 제대로 가져오지 못했을 가능성이 큽니다.

해석: 매출은 서울특별시가 570,482,124로 가장 크고, 경기도가 530,726,511로 두 번째다. 주문 건수도 두 지역이 가장 많다. 이 결과는 고객 차원의 `region` 필드가 주문 팩트에 정상적으로 필터를 전달하고 있음을 보여 준다.

## 강사용 검증 과정

이 차시는 Power BI 차시이지만, 정답 숫자는 별도의 강사용 Python 검증 코드로 원본 데이터와 맞춰 보았습니다. 학생이 따라 입력해야 하는 부분은 아니고, Power BI에서 나온 값이 실제 데이터 기준과 같은지 확인하기 위한 검산 과정입니다.

코드가 검증하는 내용은 아래 항목들입니다.

- `orders`, `customers`, `products`의 실제 행 수와 열 수
- `customers[customer_id]`, `products[product_id]`의 고유성
- `orders`의 고객번호와 상품번호가 차원 테이블에 모두 존재하는지
- 날짜 차원이 2024-01-01부터 2024-12-31까지 366행인지
- 전체 매출과 지역별 매출이 Power BI 검산값과 일치하는지

실행 확인 결과 요약:

```text
orders: 20,000행, 6열
customers: 3,000행, 7열
products: 200행, 5열
orders -> customers 참조 누락: 0
orders -> products 참조 누락: 0
날짜테이블: 366행
전체 주문 건수: 20,000
전체 매출: 1,938,489,803
지역 수: 17
S16 검증 통과
```
