# S08. 좋은 분석 프롬프트 설계

## 차시 개요

- 대상: 엑셀은 쓰지만 데이터분석은 처음인 비개발자
- 선수 차시: S07
- 난이도: 3
- 시간: 50분
- 주 데이터: 수업 홈의 데이터 받기 페이지에서 내려받은 고객 설문 정리본
- 보조 데이터: 수업 홈의 데이터 받기 페이지에서 내려받은 고객 설문 점검본
- 선택 데이터: 수업 홈의 데이터 받기 페이지에서 내려받은 영화 데이터 확장 실습 파일
- 확인된 데이터 규모: 설문 정리본 1,500행 × 8열, 설문 점검본 1,510행 × 8열, 영화 데이터 600행 × 12열

AI에게 "분석해줘"라고만 말하면, AI는 무엇을 봐야 하는지 스스로 짐작해서 답합니다. 겉으로는 그럴듯한 문장이 나오지만, 우리가 업무에서 확인하고 다시 쓸 수 있는 분석 결과가 되기는 어렵습니다. 피벗테이블을 만들 때 행, 열, 값 영역을 정해 주듯이 AI에게도 데이터의 맥락, 알고 싶은 질문, 결과를 받을 모양을 알려 주어야 합니다.

프롬프트는 AI에게 입력하는 요청문입니다. 이 차시에서는 프롬프트를 단순한 질문이 아니라 "분석 요청서"처럼 써 봅니다. 요청서가 분명해야 AI 답변도 원본과 대조할 수 있고, 엑셀에서 다시 집계할 수 있는 표로 돌아옵니다.

## 학습 목표

이 차시가 끝나면 여러분은 아래 일을 할 수 있습니다.

1. 좋은 분석 프롬프트의 세 요소인 맥락, 질문, 출력형식을 설명할 수 있습니다.
2. 한 번에 큰 답을 요구하지 않고 코드북 설계, 행별 분류, 요약, 검증 순서로 프롬프트를 심화할 수 있습니다.
3. 설문 주관식 응답을 주제와 감성으로 분류하도록 AI에게 요청할 수 있습니다.
4. 같은 데이터에 대해 프롬프트 3개를 비교하고 어떤 프롬프트가 더 재현 가능한지 판단할 수 있습니다.

여기서 재현 가능하다는 말은 같은 조건으로 다시 요청했을 때 비슷한 기준과 비슷한 모양의 결과를 얻을 수 있다는 뜻입니다. 분석은 한 번 멋있게 말하는 것보다, 나중에 다시 확인해도 기준이 흔들리지 않는 것이 더 중요합니다.

## 데이터 이해

기본 실습에서는 수업 홈의 데이터 받기 페이지에서 내려받은 고객 설문 정리본을 사용합니다. 파일을 받을 때는 엑셀 파일(.xlsx)을 선택해도 괜찮습니다. AI 파일 업로드나 파이썬 확인이 필요한 상황에서는 엑셀로도 열리는 표 파일인 CSV를 받아 쓰면 됩니다.

| 컬럼 | 의미 | 이번 차시에서 쓰는 방식 |
|---|---|---|
| `respondent_id` | 응답자 ID | AI 결과와 원본을 대조하는 기준 |
| `age_group` | 연령대 | 요약 해석의 보조 정보 |
| `gender` | 성별 | 이번 차시에서는 주요 분석 기준으로 쓰지 않음 |
| `nps` | 추천 의향 점수, 0-10 | 비추천, 중립, 추천 그룹 구분 |
| `satisfaction` | 만족도, 1-5 | 감성 해석의 참고값 |
| `channel` | 유입 또는 응답 채널 | 채널별 차이 해석의 보조 정보 |
| `free_text` | 주관식 응답 | 주제분류와 감성요약의 핵심 |
| `submitted_at` | 응답 제출 시각 | 이번 차시에서는 사용하지 않음 |

이 표에서 가장 자주 보게 될 컬럼은 `free_text`입니다. 고객이 직접 쓴 문장이 들어 있는 칸이기 때문입니다. `nps`는 추천 의향 점수입니다. 0-10 사이의 점수로, 고객이 이 서비스를 다른 사람에게 추천할 마음이 얼마나 있는지 나타냅니다. `satisfaction`은 만족도이며 1-5 사이의 값입니다. 숫자는 고객의 태도를 빠르게 보는 데 도움이 되고, 주관식 문장은 왜 그런 태도가 나왔는지 살펴보는 데 도움이 됩니다.

보조로 보는 고객 설문 점검본은 1,510행입니다. 실제 확인 결과 중복 응답자 ID 10건, NPS 결측 60건, 주관식 결측 22건, 만족도 문자 혼입 75건, `20s`와 `이십대` 같은 20대 표기 혼재 90건이 있습니다. 본 실습은 정리본으로 진행합니다. 점검본은 데이터 상태를 AI에게 알려 주지 않으면 어떤 문제가 생기는지 설명할 때 사용합니다. 예를 들어 만족도 칸에 숫자가 아니라 문자가 섞여 있으면, AI나 엑셀이 평균을 계산할 때 엉뚱하게 처리할 수 있습니다.

선택으로 보는 영화 데이터 확장 실습 파일은 600행 × 12열이며 `genre` 컬럼이 이미 들어 있습니다. 시간이 남으면 "이미 사람이 만든 범주가 있는 데이터"와 "AI에게 범주를 만들게 하는 설문 데이터"를 비교하는 확장 예제로만 사용합니다. 영화 데이터에서는 장르라는 이름표가 이미 붙어 있지만, 설문 주관식 응답에서는 우리가 주제 이름표를 만들어야 한다는 차이가 있습니다.

## 꼭 이해할 개념

### 1. 프롬프트는 AI에게 보내는 분석 요청서입니다

입문자가 흔히 쓰는 프롬프트는 "이 데이터 분석해줘"입니다. 이렇게 말하면 AI는 데이터에서 뭔가를 골라 설명해 주기는 합니다. 하지만 그 설명이 우리가 정말 알고 싶었던 질문에 답하는지는 알기 어렵습니다. 피벗테이블을 만들면서 행과 열과 값 영역을 비워 둔 것과 비슷합니다. 엑셀이 계산을 못 하는 것이 아니라, 우리가 무엇을 계산할지 지정하지 않은 상태인 거죠.

분석 프롬프트에는 최소한 세 가지가 들어가야 합니다.

| 요소 | 뜻 | 설문 분석 예시 |
|---|---|---|
| 맥락 | 이 데이터가 무엇이고 왜 보는지 | 온라인 쇼핑 고객 설문이며, 주관식 불만과 칭찬을 운영 개선에 쓰려 한다 |
| 질문 | 무엇을 알고 싶은지 | `free_text`를 읽고 주제와 감성을 분류하라 |
| 출력형식 | 어떤 모양으로 돌려받을지 | `respondent_id`, 주제, 감성, 근거문구, 신뢰도 표로 작성하라 |

좋은 프롬프트는 AI를 똑똑하게 만드는 주문이 아닙니다. 분석 기준을 좁히고, 나중에 사람이 검토할 수 있는 표로 결과를 받기 위한 업무 지시서입니다. 그래서 멋진 표현보다 분명한 조건이 더 중요합니다.

### 2. 맥락은 AI가 기준을 세우는 재료입니다

맥락은 "이 데이터가 어떤 상황에서 나온 것인지"를 알려 주는 설명입니다. 같은 "배송이 늦었다"라는 문장도 분석 목적에 따라 다르게 읽힙니다. 고객경험 개선 보고서라면 배송 프로세스 문제로 볼 수 있습니다. 마케팅 보고서라면 구매 후 안내나 기대 관리의 문제로 볼 수도 있습니다.

그래서 프롬프트에는 아래 맥락을 적어야 합니다.

- 데이터의 출처와 단위: 고객 설문 1행이 응답 1건임
- 주요 컬럼의 뜻: `nps`, `satisfaction`, `free_text`
- 분석 목적: 운영 개선 아이디어를 찾기 위함
- 제한 조건: 모르면 추측하지 말고 "판단 어려움"으로 표시
- 검증 기준: 결과에는 원문 근거와 응답자 ID를 포함

엑셀로 치면 맥락은 표 제목이나 컬럼 설명에 해당합니다. 제목 없이 숫자만 늘어놓으면 그 숫자가 매출인지 수량인지 알 수 없습니다. AI도 마찬가지입니다. 데이터가 무엇을 뜻하는지 알려 주어야 판단 기준을 제대로 세울 수 있습니다.

### 3. 질문은 한 번에 하나씩 좁혀야 합니다

"고객 반응을 분석해줘"는 너무 넓은 질문입니다. 고객 반응에는 불만, 칭찬, 사용성, 가격, 배송, 고객지원 같은 여러 이야기가 섞여 있을 수 있습니다. 이런 요청을 한 번에 던지면 AI는 자기 방식대로 기준을 만들고 바로 요약까지 해 버립니다. 그러면 나중에 왜 그런 결론이 나왔는지 되짚기가 어렵습니다.

좋은 질문은 단계별로 좁혀 갑니다.

1. 이 설문 주관식을 분류할 주제 코드 6-8개를 먼저 제안해줘.
2. 제안한 코드북으로 표본 18행을 주제와 감성으로 분류해줘.
3. 분류 결과를 주제별 응답수, 평균 NPS, 대표 근거로 요약해줘.
4. 모호하거나 다시 검토해야 할 행을 따로 표시해줘.

코드북은 분류 기준표입니다. 도서관에서 책을 주제별로 꽂기 전에 "문학", "역사", "과학" 같은 분류 이름을 정하는 것과 비슷합니다. 먼저 기준표를 만들고, 그 기준표로 행을 분류한 다음, 요약은 마지막에 하는 흐름이 안정적입니다.

### 4. 출력형식은 검증 가능하게 지정합니다

AI가 줄글로만 답하면 읽기에는 편합니다. 하지만 검증하기는 어렵습니다. 어떤 응답자를 보고 그런 판단을 했는지, 어떤 원문을 근거로 삼았는지 다시 찾아보기 힘들기 때문입니다. 그래서 이 차시에서는 결과를 표로 받는 방식을 기본으로 삼습니다.

권장 출력 컬럼을 보겠습니다.

| 출력 컬럼 | 이유 |
|---|---|
| `respondent_id` | 원본 행과 대조하기 위해 필요 |
| `topic` | 주관식 응답의 주요 주제 |
| `sentiment` | 긍정, 부정, 혼합, 중립 |
| `evidence` | 판단 근거가 된 원문 일부 |
| `confidence` | 높음, 중간, 낮음 |
| `needs_review` | 사람이 다시 봐야 할 행 표시 |

`respondent_id`는 원본과 AI 결과를 이어 주는 이름표입니다. `evidence`는 AI가 왜 그렇게 판단했는지 보여 주는 근거 문구입니다. `needs_review`는 사람이 다시 봐야 할 행을 표시하는 칸입니다. 이 열들이 있어야 AI 답변을 그대로 믿지 않고, 원본과 맞춰 보며 확인할 수 있습니다.

엑셀로 치면 출력형식은 피벗테이블 결과를 어디에 어떤 모양으로 놓을지 정하는 일입니다. 표로 받아 두면 엑셀에 그대로 붙여 넣어 피벗테이블이나 필터, 정렬로 이어 갈 수 있습니다.

### 5. 범주분류와 감성분석은 "초안"으로 받아야 합니다

범주분류는 비슷한 의견끼리 같은 이름표를 붙이는 일입니다. 예를 들어 배송 불만은 배송으로, 결제 오류는 결제로 묶는 식입니다. 감성분석은 문장의 분위기를 긍정, 부정, 혼합, 중립처럼 나누어 보는 일입니다.

주관식 응답은 숫자처럼 정답이 하나로 떨어지는 데이터가 아닙니다. "사용 중 오류가 있었지만 재접속 후 해결됐다"는 응답에는 앱 오류라는 부정 신호와 해결 경험이라는 긍정 신호가 함께 들어 있습니다. 이런 문장을 억지로 긍정이나 부정 한쪽에만 밀어 넣으면 분석이 거칠어집니다.

따라서 프롬프트에는 아래 조건을 넣습니다.

- 주제는 하나만 고르되, 애매하면 `needs_review=예`로 표시한다.
- 감성은 긍정, 부정, 혼합, 중립 중 하나로 정한다.
- NPS와 주관식 문장이 어긋나면 혼합 또는 검토 필요로 표시한다.
- 각 판단에는 원문 근거를 붙인다.

AI가 잘하는 일은 텍스트를 빠르게 훑고 분류 초안을 만드는 것입니다. 최종 판단까지 모두 맡기는 도구로 보면 위험합니다. 샘플을 사람이 검토하고, 기준표를 고치고, 원본과 대조하는 과정이 뒤따라야 업무에 쓸 수 있는 분석이 됩니다.

## 엑셀 기능 비교

엑셀과 AI는 서로 경쟁하는 도구라기보다 역할이 다릅니다. 엑셀은 정해진 숫자와 범주를 계산하고 집계하는 데 강합니다. AI는 주관식 문장을 읽고 임시 분류 기준을 만들거나 대표 근거를 뽑는 데 도움이 됩니다.

| 하고 싶은 일 | 엑셀 중심 접근 | AI 프롬프트 접근 |
|---|---|---|
| NPS 구간 나누기 | `IF`, `IFS` 수식 | 기준을 알려주고 구간명을 붙이게 함 |
| 이미 있는 범주 집계 | 피벗테이블 | 요약표 요청 가능 |
| 주관식 의견 묶기 | 사람이 읽고 새 열에 직접 입력 | 주제 코드북을 만들고 행별 분류 요청 |
| 불만 원인 요약 | 필터 후 직접 읽기 | 부정 또는 혼합 응답의 대표 근거 요청 |
| 결과 검증 | 원본 행과 대조 | `respondent_id`, 근거문구, 검토 필요 열을 요구 |

AI가 엑셀을 대체하는 것은 아닙니다. AI는 주관식 텍스트를 읽어 초안을 뽑고, 엑셀은 그 초안을 다시 집계하고 검증하는 자리에 섭니다. 키를 잴 때 줄자가 필요하고, 기록을 비교할 때 표가 필요한 것처럼, AI와 엑셀은 서로 다른 일을 맡습니다.

## 50분 수업 흐름

50분 동안은 짧은 프롬프트와 구조화된 프롬프트를 직접 비교합니다. 같은 데이터라도 요청 방식이 달라지면 결과의 검토 가능성이 얼마나 달라지는지 확인해 봅니다.

| 시간 | 내용 | 진행 방식 |
|---:|---|---|
| 0-5분 | 도입: "분석해줘"가 왜 부족한가 | 같은 데이터에 짧은 프롬프트와 구조화 프롬프트를 비교 |
| 5-12분 | 개념: 맥락, 질문, 출력형식 | 엑셀 피벗테이블의 행, 열, 값과 비교 |
| 12-24분 | 따라하기 1: 설문 표본 18행으로 주제분류 요청 | 파일 업로드 또는 표본 복사 붙여넣기 |
| 24-34분 | 따라하기 2: 단계적 심화 | 코드북 제안, 행별 분류, 요약, 검토 필요 행 요청 |
| 34-45분 | 직접 해보기: 프롬프트 3버전 A/B/C 비교 | 수강생이 직접 프롬프트 개선 |
| 45-50분 | 정리와 체크포인트 | 좋은 프롬프트의 세 요소와 검증 습관 확인 |

수업 흐름에서 중요한 부분은 비교입니다. 프롬프트 A, B, C를 따로 만들어 보면 어떤 요청이 기준을 잘 남기고, 어떤 요청이 원본 대조를 어렵게 만드는지 눈에 보입니다.

## 수업 중 강조할 원칙

1. AI에게 맡기기 전에 분석 목적을 먼저 정합니다.
2. 프롬프트에는 맥락, 질문, 출력형식이 모두 들어가야 합니다.
3. 큰 질문은 코드북 설계, 행별 분류, 요약, 검증 순서로 나눕니다.
4. 주관식 분류 결과는 정답이 아니라 검토 가능한 초안입니다.
5. AI 답변에는 원문 근거와 응답자 ID가 있어야 다시 확인할 수 있습니다.

여기서 가장 조심할 점은 AI 답변을 최종 결론처럼 받아들이지 않는 것입니다. AI가 표를 만들어 주면 일이 끝난 것처럼 보이지만, 실제로는 검토할 수 있는 첫 번째 표가 생긴 상태입니다. 원문 근거를 보고, 점수와 문장이 어긋나는 행을 찾아보고, 분류 기준이 너무 넓거나 좁지 않은지 확인해 보세요.

## 차시 평가 기준

- 상: 같은 표본 또는 내려받은 데이터로 프롬프트 3개를 비교하고, 가장 좋은 프롬프트를 근거와 함께 설명하며, 주제·감성 분류표를 검토 가능한 형식으로 만듭니다.
- 중: 따라하기 프롬프트를 재현하고, 강사의 안내를 받아 출력형식과 근거 열을 포함한 결과를 얻습니다.
- 하: "분석해줘" 수준의 프롬프트에서 벗어나지 못하거나, AI 결과를 원본과 대조하지 못합니다.
