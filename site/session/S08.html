S08. 좋은 분석 프롬프트 설계

차시 개요

  • 대상: 엑셀은 쓰지만 데이터분석은 처음인 비개발자
  • 선수 차시: S07
  • 난이도: 3
  • 시간: 50분
  • 주 데이터: 수업 홈의 데이터 받기 페이지에서 내려받은 고객 설문 정리본
  • 보조 데이터: 수업 홈의 데이터 받기 페이지에서 내려받은 고객 설문 점검본
  • 선택 데이터: 수업 홈의 데이터 받기 페이지에서 내려받은 영화 데이터 확장 실습 파일
  • 확인된 데이터 규모: 설문 정리본 1,500행 × 8열, 설문 점검본 1,510행 × 8열, 영화 데이터 600행 × 12열

AI에게 "분석해줘"라고만 말하면, AI는 무엇을 봐야 하는지 스스로 짐작해서 답합니다. 겉으로는 그럴듯한 문장이 나오지만, 우리가 업무에서 확인하고 다시 쓸 수 있는 분석 결과가 되기는 어렵습니다. 피벗테이블을 만들 때 행, 열, 값 영역을 정해 주듯이 AI에게도 데이터의 맥락, 알고 싶은 질문, 결과를 받을 모양을 알려 주어야 합니다.

프롬프트는 AI에게 입력하는 요청문입니다. 이 차시에서는 프롬프트를 단순한 질문이 아니라 "분석 요청서"처럼 써 봅니다. 요청서가 분명해야 AI 답변도 원본과 대조할 수 있고, 엑셀에서 다시 집계할 수 있는 표로 돌아옵니다.

학습 목표

이 차시가 끝나면 여러분은 아래 일을 할 수 있습니다.

  1. 좋은 분석 프롬프트의 세 요소인 맥락, 질문, 출력형식을 설명할 수 있습니다.
  2. 한 번에 큰 답을 요구하지 않고 코드북 설계, 행별 분류, 요약, 검증 순서로 프롬프트를 심화할 수 있습니다.
  3. 설문 주관식 응답을 주제와 감성으로 분류하도록 AI에게 요청할 수 있습니다.
  4. 같은 데이터에 대해 프롬프트 3개를 비교하고 어떤 프롬프트가 더 재현 가능한지 판단할 수 있습니다.

여기서 재현 가능하다는 말은 같은 조건으로 다시 요청했을 때 비슷한 기준과 비슷한 모양의 결과를 얻을 수 있다는 뜻입니다. 분석은 한 번 멋있게 말하는 것보다, 나중에 다시 확인해도 기준이 흔들리지 않는 것이 더 중요합니다.

데이터 이해

기본 실습에서는 수업 홈의 데이터 받기 페이지에서 내려받은 고객 설문 정리본을 사용합니다. 파일을 받을 때는 엑셀 파일(.xlsx)을 선택해도 괜찮습니다. AI 파일 업로드나 파이썬 확인이 필요한 상황에서는 엑셀로도 열리는 표 파일인 CSV를 받아 쓰면 됩니다.

컬럼 의미 이번 차시에서 쓰는 방식
respondent_id 응답자 ID AI 결과와 원본을 대조하는 기준
age_group 연령대 요약 해석의 보조 정보
gender 성별 이번 차시에서는 주요 분석 기준으로 쓰지 않음
nps 추천 의향 점수, 0-10 비추천, 중립, 추천 그룹 구분
satisfaction 만족도, 1-5 감성 해석의 참고값
channel 유입 또는 응답 채널 채널별 차이 해석의 보조 정보
free_text 주관식 응답 주제분류와 감성요약의 핵심
submitted_at 응답 제출 시각 이번 차시에서는 사용하지 않음

이 표에서 가장 자주 보게 될 컬럼은 free_text입니다. 고객이 직접 쓴 문장이 들어 있는 칸이기 때문입니다. nps는 추천 의향 점수입니다. 0-10 사이의 점수로, 고객이 이 서비스를 다른 사람에게 추천할 마음이 얼마나 있는지 나타냅니다. satisfaction은 만족도이며 1-5 사이의 값입니다. 숫자는 고객의 태도를 빠르게 보는 데 도움이 되고, 주관식 문장은 왜 그런 태도가 나왔는지 살펴보는 데 도움이 됩니다.

보조로 보는 고객 설문 점검본은 1,510행입니다. 실제 확인 결과 중복 응답자 ID 10건, NPS 결측 60건, 주관식 결측 22건, 만족도 문자 혼입 75건, 20s와 이십대 같은 20대 표기 혼재 90건이 있습니다. 본 실습은 정리본으로 진행합니다. 점검본은 데이터 상태를 AI에게 알려 주지 않으면 어떤 문제가 생기는지 설명할 때 사용합니다. 예를 들어 만족도 칸에 숫자가 아니라 문자가 섞여 있으면, AI나 엑셀이 평균을 계산할 때 엉뚱하게 처리할 수 있습니다.

선택으로 보는 영화 데이터 확장 실습 파일은 600행 × 12열이며 genre 컬럼이 이미 들어 있습니다. 시간이 남으면 "이미 사람이 만든 범주가 있는 데이터"와 "AI에게 범주를 만들게 하는 설문 데이터"를 비교하는 확장 예제로만 사용합니다. 영화 데이터에서는 장르라는 이름표가 이미 붙어 있지만, 설문 주관식 응답에서는 우리가 주제 이름표를 만들어야 한다는 차이가 있습니다.

꼭 이해할 개념

1. 프롬프트는 AI에게 보내는 분석 요청서입니다

입문자가 흔히 쓰는 프롬프트는 "이 데이터 분석해줘"입니다. 이렇게 말하면 AI는 데이터에서 뭔가를 골라 설명해 주기는 합니다. 하지만 그 설명이 우리가 정말 알고 싶었던 질문에 답하는지는 알기 어렵습니다. 피벗테이블을 만들면서 행과 열과 값 영역을 비워 둔 것과 비슷합니다. 엑셀이 계산을 못 하는 것이 아니라, 우리가 무엇을 계산할지 지정하지 않은 상태인 거죠.

분석 프롬프트에는 최소한 세 가지가 들어가야 합니다.

요소 뜻 설문 분석 예시
맥락 이 데이터가 무엇이고 왜 보는지 온라인 쇼핑 고객 설문이며, 주관식 불만과 칭찬을 운영 개선에 쓰려 한다
질문 무엇을 알고 싶은지 free_text를 읽고 주제와 감성을 분류하라
출력형식 어떤 모양으로 돌려받을지 respondent_id, 주제, 감성, 근거문구, 신뢰도 표로 작성하라

좋은 프롬프트는 AI를 똑똑하게 만드는 주문이 아닙니다. 분석 기준을 좁히고, 나중에 사람이 검토할 수 있는 표로 결과를 받기 위한 업무 지시서입니다. 그래서 멋진 표현보다 분명한 조건이 더 중요합니다.

2. 맥락은 AI가 기준을 세우는 재료입니다

맥락은 "이 데이터가 어떤 상황에서 나온 것인지"를 알려 주는 설명입니다. 같은 "배송이 늦었다"라는 문장도 분석 목적에 따라 다르게 읽힙니다. 고객경험 개선 보고서라면 배송 프로세스 문제로 볼 수 있습니다. 마케팅 보고서라면 구매 후 안내나 기대 관리의 문제로 볼 수도 있습니다.

그래서 프롬프트에는 아래 맥락을 적어야 합니다.

  • 데이터의 출처와 단위: 고객 설문 1행이 응답 1건임
  • 주요 컬럼의 뜻: nps, satisfaction, free_text
  • 분석 목적: 운영 개선 아이디어를 찾기 위함
  • 제한 조건: 모르면 추측하지 말고 "판단 어려움"으로 표시
  • 검증 기준: 결과에는 원문 근거와 응답자 ID를 포함

엑셀로 치면 맥락은 표 제목이나 컬럼 설명에 해당합니다. 제목 없이 숫자만 늘어놓으면 그 숫자가 매출인지 수량인지 알 수 없습니다. AI도 마찬가지입니다. 데이터가 무엇을 뜻하는지 알려 주어야 판단 기준을 제대로 세울 수 있습니다.

3. 질문은 한 번에 하나씩 좁혀야 합니다

"고객 반응을 분석해줘"는 너무 넓은 질문입니다. 고객 반응에는 불만, 칭찬, 사용성, 가격, 배송, 고객지원 같은 여러 이야기가 섞여 있을 수 있습니다. 이런 요청을 한 번에 던지면 AI는 자기 방식대로 기준을 만들고 바로 요약까지 해 버립니다. 그러면 나중에 왜 그런 결론이 나왔는지 되짚기가 어렵습니다.

좋은 질문은 단계별로 좁혀 갑니다.

  1. 이 설문 주관식을 분류할 주제 코드 6-8개를 먼저 제안해줘.
  2. 제안한 코드북으로 표본 18행을 주제와 감성으로 분류해줘.
  3. 분류 결과를 주제별 응답수, 평균 NPS, 대표 근거로 요약해줘.
  4. 모호하거나 다시 검토해야 할 행을 따로 표시해줘.

코드북은 분류 기준표입니다. 도서관에서 책을 주제별로 꽂기 전에 "문학", "역사", "과학" 같은 분류 이름을 정하는 것과 비슷합니다. 먼저 기준표를 만들고, 그 기준표로 행을 분류한 다음, 요약은 마지막에 하는 흐름이 안정적입니다.

4. 출력형식은 검증 가능하게 지정합니다

AI가 줄글로만 답하면 읽기에는 편합니다. 하지만 검증하기는 어렵습니다. 어떤 응답자를 보고 그런 판단을 했는지, 어떤 원문을 근거로 삼았는지 다시 찾아보기 힘들기 때문입니다. 그래서 이 차시에서는 결과를 표로 받는 방식을 기본으로 삼습니다.

권장 출력 컬럼을 보겠습니다.

출력 컬럼 이유
respondent_id 원본 행과 대조하기 위해 필요
topic 주관식 응답의 주요 주제
sentiment 긍정, 부정, 혼합, 중립
evidence 판단 근거가 된 원문 일부
confidence 높음, 중간, 낮음
needs_review 사람이 다시 봐야 할 행 표시

respondent_id는 원본과 AI 결과를 이어 주는 이름표입니다. evidence는 AI가 왜 그렇게 판단했는지 보여 주는 근거 문구입니다. needs_review는 사람이 다시 봐야 할 행을 표시하는 칸입니다. 이 열들이 있어야 AI 답변을 그대로 믿지 않고, 원본과 맞춰 보며 확인할 수 있습니다.

엑셀로 치면 출력형식은 피벗테이블 결과를 어디에 어떤 모양으로 놓을지 정하는 일입니다. 표로 받아 두면 엑셀에 그대로 붙여 넣어 피벗테이블이나 필터, 정렬로 이어 갈 수 있습니다.

5. 범주분류와 감성분석은 "초안"으로 받아야 합니다

범주분류는 비슷한 의견끼리 같은 이름표를 붙이는 일입니다. 예를 들어 배송 불만은 배송으로, 결제 오류는 결제로 묶는 식입니다. 감성분석은 문장의 분위기를 긍정, 부정, 혼합, 중립처럼 나누어 보는 일입니다.

주관식 응답은 숫자처럼 정답이 하나로 떨어지는 데이터가 아닙니다. "사용 중 오류가 있었지만 재접속 후 해결됐다"는 응답에는 앱 오류라는 부정 신호와 해결 경험이라는 긍정 신호가 함께 들어 있습니다. 이런 문장을 억지로 긍정이나 부정 한쪽에만 밀어 넣으면 분석이 거칠어집니다.

따라서 프롬프트에는 아래 조건을 넣습니다.

  • 주제는 하나만 고르되, 애매하면 needs_review=예로 표시한다.
  • 감성은 긍정, 부정, 혼합, 중립 중 하나로 정한다.
  • NPS와 주관식 문장이 어긋나면 혼합 또는 검토 필요로 표시한다.
  • 각 판단에는 원문 근거를 붙인다.

AI가 잘하는 일은 텍스트를 빠르게 훑고 분류 초안을 만드는 것입니다. 최종 판단까지 모두 맡기는 도구로 보면 위험합니다. 샘플을 사람이 검토하고, 기준표를 고치고, 원본과 대조하는 과정이 뒤따라야 업무에 쓸 수 있는 분석이 됩니다.

엑셀 기능 비교

엑셀과 AI는 서로 경쟁하는 도구라기보다 역할이 다릅니다. 엑셀은 정해진 숫자와 범주를 계산하고 집계하는 데 강합니다. AI는 주관식 문장을 읽고 임시 분류 기준을 만들거나 대표 근거를 뽑는 데 도움이 됩니다.

하고 싶은 일 엑셀 중심 접근 AI 프롬프트 접근
NPS 구간 나누기 IF, IFS 수식 기준을 알려주고 구간명을 붙이게 함
이미 있는 범주 집계 피벗테이블 요약표 요청 가능
주관식 의견 묶기 사람이 읽고 새 열에 직접 입력 주제 코드북을 만들고 행별 분류 요청
불만 원인 요약 필터 후 직접 읽기 부정 또는 혼합 응답의 대표 근거 요청
결과 검증 원본 행과 대조 respondent_id, 근거문구, 검토 필요 열을 요구

AI가 엑셀을 대체하는 것은 아닙니다. AI는 주관식 텍스트를 읽어 초안을 뽑고, 엑셀은 그 초안을 다시 집계하고 검증하는 자리에 섭니다. 키를 잴 때 줄자가 필요하고, 기록을 비교할 때 표가 필요한 것처럼, AI와 엑셀은 서로 다른 일을 맡습니다.

50분 수업 흐름

50분 동안은 짧은 프롬프트와 구조화된 프롬프트를 직접 비교합니다. 같은 데이터라도 요청 방식이 달라지면 결과의 검토 가능성이 얼마나 달라지는지 확인해 봅니다.

시간 내용 진행 방식
0-5분 도입: "분석해줘"가 왜 부족한가 같은 데이터에 짧은 프롬프트와 구조화 프롬프트를 비교
5-12분 개념: 맥락, 질문, 출력형식 엑셀 피벗테이블의 행, 열, 값과 비교
12-24분 따라하기 1: 설문 표본 18행으로 주제분류 요청 파일 업로드 또는 표본 복사 붙여넣기
24-34분 따라하기 2: 단계적 심화 코드북 제안, 행별 분류, 요약, 검토 필요 행 요청
34-45분 직접 해보기: 프롬프트 3버전 A/B/C 비교 수강생이 직접 프롬프트 개선
45-50분 정리와 체크포인트 좋은 프롬프트의 세 요소와 검증 습관 확인

수업 흐름에서 중요한 부분은 비교입니다. 프롬프트 A, B, C를 따로 만들어 보면 어떤 요청이 기준을 잘 남기고, 어떤 요청이 원본 대조를 어렵게 만드는지 눈에 보입니다.

수업 중 강조할 원칙

  1. AI에게 맡기기 전에 분석 목적을 먼저 정합니다.
  2. 프롬프트에는 맥락, 질문, 출력형식이 모두 들어가야 합니다.
  3. 큰 질문은 코드북 설계, 행별 분류, 요약, 검증 순서로 나눕니다.
  4. 주관식 분류 결과는 정답이 아니라 검토 가능한 초안입니다.
  5. AI 답변에는 원문 근거와 응답자 ID가 있어야 다시 확인할 수 있습니다.

여기서 가장 조심할 점은 AI 답변을 최종 결론처럼 받아들이지 않는 것입니다. AI가 표를 만들어 주면 일이 끝난 것처럼 보이지만, 실제로는 검토할 수 있는 첫 번째 표가 생긴 상태입니다. 원문 근거를 보고, 점수와 문장이 어긋나는 행을 찾아보고, 분류 기준이 너무 넓거나 좁지 않은지 확인해 보세요.

차시 평가 기준

  • 상: 같은 표본 또는 내려받은 데이터로 프롬프트 3개를 비교하고, 가장 좋은 프롬프트를 근거와 함께 설명하며, 주제·감성 분류표를 검토 가능한 형식으로 만듭니다.
  • 중: 따라하기 프롬프트를 재현하고, 강사의 안내를 받아 출력형식과 근거 열을 포함한 결과를 얻습니다.
  • 하: "분석해줘" 수준의 프롬프트에서 벗어나지 못하거나, AI 결과를 원본과 대조하지 못합니다.

S08. 좋은 분석 프롬프트 설계

사용할 데이터

좋은 분석 프롬프트는 AI에게 보내는 분석 요청서입니다. 수업에서는 고객 설문 응답을 가지고, AI가 주관식 문장을 어떻게 주제와 감성으로 나누는지 직접 확인해 봅니다.

기본 실습은 수업 홈의 데이터 받기 페이지에서 내려받은 고객 설문 정제 파일로 진행합니다. CSV는 엑셀로도 열리는 표 파일입니다. 엑셀 파일이 더 익숙한 분은 같은 데이터의 엑셀 파일을 받아 사용해도 됩니다.

보조 확인용으로는 데이터 상태가 일부러 조금 섞여 있는 고객 설문 파일을 봅니다. 선택 확장용으로는 영화 장르 데이터 파일을 사용할 수 있습니다. 선택 확장 파일은 시간이 남을 때만 다룹니다.

AI 파일 업로드 기능은 쓰는 도구와 요금제에 따라 막혀 있을 수 있습니다. 업로드가 되면 데이터 받기 페이지에서 내려받은 고객 설문 정제 파일을 올리고, 업로드가 안 되면 아래 표본 18행을 복사해서 씁니다. 분석 재료를 먼저 분명히 해야 AI 답변도 어디까지 믿을 수 있는지 판단할 수 있습니다.

오늘의 목표

오늘은 AI에게 설문 주관식 응답을 분석시켜 봅니다. 다만 목표는 AI 답변을 그대로 믿는 것이 아닙니다. 좋은 분석 프롬프트를 설계해서, 사람이 다시 확인할 수 있는 표 형태의 결과를 받아 내는 연습을 합니다.

좋은 분석 프롬프트에는 세 가지가 들어갑니다. 엑셀에서 피벗테이블을 만들 때 행, 열, 값을 지정하듯이 AI에게도 기준을 알려 주는 겁니다.

요소 확인 질문
맥락 이 데이터가 무엇이고 왜 분석하는가
질문 AI가 정확히 무엇을 해야 하는가
출력형식 결과를 어떤 표와 문장으로 받을 것인가

맥락은 “이 표가 어떤 표인지” 알려 주는 설명입니다. 질문은 “무엇을 해 달라는지”를 좁히는 말입니다. 출력형식은 “결과를 어떤 모양으로 받을지” 정하는 약속입니다.

1단계. 파일 또는 표본 준비

먼저 AI에게 줄 재료를 정합니다. 전체 파일을 올릴지, 표본 18행만 붙여 넣을지에 따라 결과의 의미가 달라지기 때문입니다.

수업 홈의 데이터 받기 페이지에서 고객 설문 정제 파일을 내려받아 준비해 보세요. 파일 업로드가 편하면 파일을 사용하고, 업로드가 막혀 있으면 위의 표본 18행을 복사하면 됩니다.

둘 중 하나를 선택합니다.

상황 진행 방법

AI 파일 업로드 불가 위의 표본 18행을 복사한다

업로드를 했다면 AI에게 파일 이름과 컬럼명을 인식했는지 먼저 확인하게 합니다. 이 확인은 AI가 다른 데이터로 착각한 채 분석을 시작하지 않게 막는 안전장치입니다.

첨부한 파일의 행 수와 컬럼명을 확인해줘.
그 다음 내가 지시하기 전까지 분석 결론은 내지 마.

2단계. 프롬프트 A 실행

이 단계에서는 일부러 짧은 프롬프트를 써 봅니다. 그래야 “분석해줘”만 말했을 때 결과가 얼마나 흔들리는지 직접 볼 수 있습니다.

아래처럼 짧게 요청합니다.

이 설문 데이터를 분석해줘.

결과를 읽고 아래 질문에 답해 보세요. AI가 그럴듯하게 말했는지보다, 사람이 다시 확인할 수 있는지 보는 것이 중요합니다.

  1. 어떤 기준으로 주제를 나눴는가?
  2. 원문 근거가 보이는가?
  3. 결과를 엑셀에 붙여 넣어 피벗테이블로 다시 집계할 수 있는가?

프롬프트 A는 대개 설명은 그럴듯합니다. 하지만 기준이 분명하지 않고, 출력형식도 일정하지 않은 경우가 많습니다. 표로 다시 쓰기 어렵다면 업무에 바로 쓰기 힘듭니다.

3단계. 프롬프트 B 실행

이번에는 AI에게 맥락, 질문, 출력형식을 함께 줍니다. 분석을 맡기는 것이 아니라, 분석 기준을 같이 정해 주는 단계입니다.

아래 프롬프트를 실행합니다. 업로드를 하지 않았다면 마지막에 표본 18행을 붙입니다.

너는 고객경험 분석가다.
데이터는 온라인 쇼핑 서비스의 고객 설문이다. 한 행은 응답 1건이다.
주요 컬럼은 respondent_id, age_group, channel, nps, satisfaction, free_text다.

목표는 free_text를 읽고 고객 의견을 운영 개선에 쓸 수 있게 정리하는 것이다.
각 응답을 주제와 감성으로 분류해줘.

주제는 6-8개로 만들되 너무 세분화하지 마라.
감성은 긍정, 부정, 혼합, 중립 중 하나로 정하라.
NPS와 free_text의 분위기가 어긋나면 혼합 또는 검토 필요로 표시하라.

출력은 표로 작성하라.
표 컬럼은 respondent_id, topic, sentiment, evidence, confidence, needs_review로 해라.
evidence에는 판단 근거가 된 원문 일부를 넣어라.

프롬프트 A와 B의 차이를 확인합니다. 특히 evidence 열이 있는지 보세요. evidence는 판단 근거가 된 원문 일부입니다. 사전에서 단어를 찾듯이, 나중에 원문과 AI 판단을 다시 대조할 수 있게 해 줍니다.

사람이 다시 검토할 행을 찾을 수 있는가

따라하기 2. 단계적으로 심화하기

한 번에 전체 분석을 요구하지 않고 세 번에 나누어 요청합니다. 먼저 분류 기준을 만들고, 그 기준으로 행을 나누고, 마지막에 요약을 받는 순서입니다. 기준 없이 바로 요약부터 받으면 나중에 “왜 이렇게 나눴지?”를 확인하기 어렵습니다.

1단계. 코드북 먼저 만들기

코드북은 분류 기준표입니다. 예를 들어 “배송”, “결제”, “앱 사용성”처럼 어떤 의견을 어느 주제로 넣을지 정해 둔 작은 사전이라고 생각하면 됩니다.

먼저 이 설문 free_text를 분류할 주제 코드북을 만들어줘.

조건:
- 주제는 6-8개로 제한한다.
- 각 주제마다 정의와 포함 예시를 적는다.
- 서로 겹치기 쉬운 주제는 구분 기준을 적는다.
- 아직 행별 분류는 하지 말고 코드북만 제안한다.

확인할 점입니다.

  • 주제가 너무 많지 않은가
  • "기타"가 남용되지 않도록 기준이 있는가
  • 배송, 앱 사용성, 결제와 혜택, 고객지원처럼 업무 조치로 이어지는 범주인가

주제가 너무 많으면 나중에 집계가 어려워집니다. 반대로 너무 넓으면 실제로 무엇을 고쳐야 할지 흐려집니다. 그래서 6-8개 정도로 제한해 보는 겁니다.

2단계. 같은 코드북으로 행별 분류하기

코드북이 괜찮다면 이어서 요청합니다. 같은 기준표를 계속 써야 행마다 판단이 덜 흔들립니다.

방금 만든 코드북을 사용해서 아래 데이터의 각 행을 분류해줘.

규칙:
- 한 행의 topic은 하나만 고른다.
- 감성은 긍정, 부정, 혼합, 중립 중 하나로 정한다.
- 판단이 애매하면 confidence를 낮음으로 두고 needs_review를 예로 표시한다.
- 원문에 없는 내용을 추측하지 않는다.

출력 표 컬럼:
respondent_id, topic, sentiment, evidence, confidence, needs_review

결과를 받은 뒤 respondent_id가 빠진 행이 없는지 확인합니다. respondent_id는 원본과 AI 결과를 이어 주는 열입니다. 이름표가 빠지면 나중에 어느 응답을 말하는지 찾기 어렵습니다.

3단계. 요약과 검토 포인트 요청하기

이 단계에서는 행별 분류 결과를 보고 요약을 받습니다. 먼저 재료가 전체 파일인지 표본 18행인지 구분해야 합니다. 전체와 표본은 말할 수 있는 범위가 다릅니다.

파일 업로드를 한 경우:

이번에는 전체 1,500행 기준으로 요약해줘.

출력:
1. 주제별 응답수와 비중
2. 주제별 평균 NPS와 평균 satisfaction
3. 부정 또는 혼합 응답이 많은 주제 3개
4. 대표 근거문구 2개씩
5. 사람이 다시 확인해야 할 분류 기준 3개

숫자는 가능한 한 표로 보여주고, 결론은 5문장 이내로 써줘.

파일 업로드가 안 된 경우:

위 표본 18행 기준으로만 요약해줘.
전체 1,500행의 결론처럼 말하지 말고, 반드시 "표본 18행 기준"이라고 표시해줘.

중요한 건 표본 분석과 전체 분석을 갈라 보는 일입니다. 표본 18행만 보고 전체 고객의 결론인 양 쓰면 안 됩니다. 작은 표본은 연습과 점검에는 좋지만, 전체 고객의 경향이라고 말하려면 전체 1,500행을 봐야 합니다.

따라하기 3. AI 답변 검증하기

AI 답변을 받은 뒤에는 검증을 시킵니다. 검증은 AI를 의심하려고 하는 일이 아니라, 사람이 다시 봐야 할 행을 빨리 찾기 위한 과정입니다.

아래 프롬프트로 AI가 만든 분류 결과를 스스로 다시 살펴보게 합니다.

방금 만든 분류 결과에서 사람이 다시 확인해야 할 가능성이 큰 행 5개를 골라줘.

기준:
- NPS는 낮지만 free_text는 긍정적으로 보이는 행
- NPS는 높지만 free_text에 불만이나 개선 요청이 있는 행
- 하나의 문장에 긍정과 부정이 함께 있는 행
- 주제를 하나로 고르기 어려운 행

출력은 respondent_id, 검토 이유, 원문 근거로 작성해줘.

이번 표본에는 다음처럼 점수와 문장이 어긋나는 사례가 있습니다.

  • R2025-001293: NPS는 2점이지만 "부담이 없었습니다"라는 긍정 표현이 있다.
  • R2025-000534: NPS는 9점이지만 알림이 너무 잦다는 불편이 있다.
  • R2025-001185: NPS는 8점이지만 결제 오류와 재시도가 언급된다.

이런 행을 집어내는 것이 좋은 프롬프트의 성과입니다. 점수만 보면 놓치고, 문장만 봐도 놓칠 수 있는 부분을 함께 보게 만들기 때문입니다.

직접 해보기. 프롬프트 3버전 A/B/C 비교

같은 파일 또는 같은 표본 18행을 사용해 프롬프트 3개를 직접 만들어 봅니다. 같은 재료를 써야 프롬프트 차이만 비교할 수 있습니다.

조건

  1. 프롬프트 A는 일부러 짧게 만든다.
  2. 프롬프트 B는 맥락, 질문, 출력형식을 넣는다.
  3. 프롬프트 C는 코드북 설계, 행별 분류, 요약, 검증 요청으로 단계화한다.
  4. 세 프롬프트의 결과를 비교한다.

프롬프트 A는 기준이 부족한 답이 어떻게 나오는지 보는 용도입니다. 프롬프트 B는 업무 지시서처럼 조건을 갖춘 요청입니다. 프롬프트 C는 분석을 한 번에 끝내지 않고, 기준 만들기부터 검증까지 나누어 진행하는 방식입니다.

제출 형식

아래 표를 채웁니다. 빈칸에는 “좋음”, “보통”, “부족”처럼 적어도 되고, 짧은 설명을 붙여도 됩니다.

항목 프롬프트 A 프롬프트 B 프롬프트 C
맥락이 충분한가
질문이 구체적인가
출력형식이 표로 고정되는가
원문 근거가 포함되는가
검토 필요 행을 찾는가
업무에 바로 쓰기 좋은가

마지막에 아래 문장으로 적습니다.

내가 선택한 최종 프롬프트는 __이며, 이유는 __이다. 특히 __ 열이 있어 원본과 대조할 수 있다.

여기서 좋은 답은 “AI가 멋진 문장을 썼다”가 아닙니다. 원본과 대조할 수 있고, 표로 다시 집계할 수 있고, 사람이 검토해야 할 행을 찾을 수 있는 답입니다.

자주 생기는 문제

AI가 전체 결론처럼 말하지만 나는 표본만 넣었다

프롬프트에 "표본 18행 기준이라고 명시하라"를 넣습니다. 표본 분석을 전체 결론처럼 말하면 과장입니다. 표본은 맛보기 한 숟가락이지, 냄비 전체를 다 확인한 것은 아닙니다.

AI가 원문에 없는 내용을 추측한다

"원문에 없는 원인, 의도, 고객 특성은 추측하지 말라"를 조건에 넣습니다. 결과에 evidence 열을 요구하면 추측을 줄일 수 있습니다. 원문 근거가 없는 말은 나중에 확인할 길이 없기 때문입니다.

주제가 너무 많아진다

"주제는 6-8개로 제한한다"를 넣습니다. 분류가 너무 세밀하면 나중에 피벗테이블로 집계하기 어렵습니다. 사전을 만들 때 단어가 너무 잘게 쪼개져 있으면 찾기 힘든 것과 비슷합니다.

같은 문장을 볼 때마다 분류가 조금씩 달라진다

먼저 코드북을 확정하고, 그 코드북을 기준으로 다시 분류하라고 요청합니다. 중요한 업무라면 표본을 사람이 검토해 코드북을 고친 뒤 전체에 적용합니다. 기준표를 고정해야 같은 문장을 볼 때 판단도 덜 흔들립니다.

개인정보가 포함된 실제 설문이면 어떻게 하나

실제 업무 데이터에는 이름, 전화번호, 이메일, 주소 같은 개인정보를 AI에 넣지 않습니다. 이번 수업 데이터는 합성 데이터지만, 실무에서는 업로드 전에 민감정보를 제거해야 합니다. 민감정보를 지운 뒤에도 분석에 필요한 열이 남아 있는지 확인하고 진행해 보세요.

강사 멘트:

"분석을 AI에게 맡길 때 제일 먼저 봐야 할 것은 버튼이 아니라 요청 문장입니다. 지난 차시까지는 데이터를 표로 정리하고, 질문에 맞춰 요약하는 연습을 했습니다. 이제는 그 분석을 AI에게 부탁할 때 어떤 말로 부탁해야 하는지 봅니다."

5-12분. 개념 설명: 맥락, 질문, 출력형식

강사 멘트:

"좋은 분석 프롬프트에는 세 가지가 들어갑니다. 맥락은 이 데이터가 무엇인지, 왜 분석하는지 알려 주는 부분입니다. 질문은 AI가 무엇을 해야 하는지 좁혀 주는 부분입니다. 출력형식은 표로 받을지, 요약문으로 받을지, 어떤 열을 넣을지 정하는 부분입니다."

"엑셀의 피벗테이블과 비교해 볼까요? 피벗테이블에서 행, 열, 값을 정하지 않으면 표가 만들어지지 않습니다. AI도 마찬가지입니다. 맥락, 질문, 출력형식이 없으면 분석 기준이 흔들립니다. 그래서 프롬프트는 길게 쓰는 글이 아니라, 분석 기준을 적어 주는 요청서라고 보면 됩니다."

판서 또는 화면 강조:

프롬프트 요소 설문 분석 예시
맥락 온라인 쇼핑 고객 설문, 운영 개선 목적
질문 주관식 응답을 주제와 감성으로 분류
출력형식 응답자 ID, 주제, 감성, 근거, 신뢰도, 검토 필요

주의해서 말할 점:

  • 프롬프트는 긴 문장이 아니라 명확한 요청서다.
  • AI 답변은 최종 정답이 아니라 검토 가능한 초안이다.
  • 원문 근거가 없는 요약은 보고서에 바로 쓰지 않는다.

12-24분. 따라하기 1: 나쁜 프롬프트와 좋은 프롬프트 비교

시연 절차:

  1. AI 도구를 열고 파일 업로드가 가능하면 수업 홈의 데이터 받기 페이지에서 내려받은 정리된 고객 설문 파일을 업로드한다.
  2. 업로드가 불가능하면 핸드아웃의 표본 18행을 복사한다.
  3. 먼저 프롬프트 A를 실행한다.

프롬프트 A:

이 설문 데이터를 분석해줘.

강사 멘트:

"이 답변은 읽기에는 자연스러울 수 있습니다. 그런데 여기서 뭐가 빠져 있을까요? 어떤 기준으로 주제를 나눴는지, 어떤 원문을 근거로 삼았는지, 결과를 엑셀에 붙여 넣어 다시 집계할 수 있는지가 분명하지 않습니다. 보고서에 쓰려면 예쁜 문장보다 확인 가능한 표가 더 중요합니다."

이어서 프롬프트 B를 실행한다.

너는 고객경험 분석가다.
데이터는 온라인 쇼핑 서비스의 고객 설문이다. 한 행은 응답 1건이다.
주요 컬럼은 respondent_id, age_group, channel, nps, satisfaction, free_text다.

목표는 free_text를 읽고 고객 의견을 운영 개선에 쓸 수 있게 정리하는 것이다.
각 응답을 주제와 감성으로 분류해줘.

주제는 6-8개로 만들되 너무 세분화하지 마라.
감성은 긍정, 부정, 혼합, 중립 중 하나로 정하라.
NPS와 free_text의 분위기가 어긋나면 혼합 또는 검토 필요로 표시하라.

출력은 표로 작성하라.
표 컬럼은 respondent_id, topic, sentiment, evidence, confidence, needs_review로 해라.
evidence에는 판단 근거가 된 원문 일부를 넣어라.
confidence는 높음, 중간, 낮음 중 하나로 표시하라.

시연 포인트:

  • 프롬프트 B의 결과에 respondent_id가 있는지 확인한다.
  • evidence 열이 원문 대조를 가능하게 한다고 설명한다.
  • needs_review는 AI가 애매한 행을 숨기지 않게 만드는 장치라고 설명한다.

확인 질문:

"프롬프트 A와 B 중 어떤 결과가 엑셀에 다시 붙여 넣기 쉬운가요?"

기대 답:

"프롬프트 B입니다. 표 형식과 열 이름이 정해져 있기 때문입니다."

24-34분. 따라하기 2: 단계적으로 깊게 요청하기

강사 멘트:

"좋은 분석은 한 번의 프롬프트로 끝나지 않습니다. 특히 주관식 텍스트는 사람이 읽는 말이기 때문에 해석 기준이 먼저 필요합니다. 먼저 분류 기준을 만들고, 그 기준으로 행을 분류하고, 마지막에 요약하는 순서가 좋습니다. 키를 잴 때도 먼저 자를 맞춰 놓고 재듯이, 텍스트 분석도 먼저 기준표를 맞춰야 합니다."

시연 1: 코드북 요청

먼저 이 설문 free_text를 분류할 주제 코드북을 만들어줘.

조건:
- 주제는 6-8개로 제한한다.
- 각 주제마다 정의와 포함 예시를 적는다.
- 서로 겹치기 쉬운 주제는 구분 기준을 적는다.
- 아직 행별 분류는 하지 말고 코드북만 제안한다.

강사 멘트:

"코드북은 분류 기준표입니다. 예를 들어 사전에서 단어 뜻을 확인하듯이, 어떤 응답을 어떤 주제로 넣을지 미리 정해 두는 표입니다. 사람이 먼저 기준을 보고 납득할 수 있어야 전체 분류를 맡길 수 있습니다. 코드북이 엉성하면 결과도 엉성해집니다."

시연 2: 행별 분류 요청

방금 만든 코드북을 사용해서 아래 데이터의 각 행을 분류해줘.

규칙:
- 한 행의 topic은 하나만 고른다.
- 감성은 긍정, 부정, 혼합, 중립 중 하나로 정한다.
- 판단이 애매하면 confidence를 낮음으로 두고 needs_review를 예로 표시한다.
- 원문에 없는 내용을 추측하지 않는다.

출력 표 컬럼:
respondent_id, topic, sentiment, evidence, confidence, needs_review

시연 3: 검토 필요 행 요청

방금 만든 분류 결과에서 사람이 다시 확인해야 할 가능성이 큰 행 5개를 골라줘.

기준:
- NPS는 낮지만 free_text는 긍정적으로 보이는 행
- NPS는 높지만 free_text에 불만이나 개선 요청이 있는 행
- 하나의 문장에 긍정과 부정이 함께 있는 행
- 주제를 하나로 고르기 어려운 행

출력은 respondent_id, 검토 이유, 원문 근거로 작성해줘.

시연 포인트:

  • R2025-001293은 NPS 2점이지만 문장은 긍정 표현이 있어 검토 대상이 될 수 있다.
  • R2025-000534는 NPS 9점이지만 알림 빈도 불만이 있어 검토 대상이 될 수 있다.
  • 점수와 문장이 항상 같은 방향으로 움직이지 않는다고 설명한다.

34-45분. 직접 해보기: 프롬프트 3버전 비교

미션 안내 멘트:

"이제 여러분이 직접 프롬프트를 세 개 만들어 비교합니다. A는 일부러 짧게 써 보세요. B는 맥락과 질문과 출력형식을 담아 보세요. C는 여러 단계로 쪼개서 만들어 봅니다. 여기서 볼 것은 어느 답변이 더 그럴싸해 보이는지가 아닙니다. 어느 답변이 원본과 대조되고, 엑셀에 다시 붙여 넣을 수 있고, 실제 업무에 다시 쓸 수 있는지를 봅니다."

수강생 작업:

  1. 같은 정리된 고객 설문 파일 또는 같은 표본 18행을 사용한다.
  2. 프롬프트 A, B, C를 각각 실행한다.
  3. 결과를 핸드아웃의 비교표에 기록한다.
  4. 최종으로 선택할 프롬프트와 이유를 한 문장으로 쓴다.

강사가 돌아다니며 볼 점:

  • 프롬프트에 데이터 맥락이 있는가
  • 주제와 감성의 정의가 있는가
  • 결과 표에 respondent_id와 evidence가 있는가
  • 표본 분석을 전체 결론처럼 쓰지 않았는가
  • 검토 필요 행을 찾도록 요청했는가

45-50분. 수업 마무리

강사 멘트:

"분석 프롬프트는 AI에게 보내는 분석 요청서입니다. 맥락과 질문과 출력형식이 갖춰져야 결과가 흔들리지 않습니다. 큰 질문은 한꺼번에 던지지 말고 코드북, 행별 분류, 요약, 검증으로 쪼개서 요청해 보세요."

"AI가 주관식 응답을 빠르게 읽어 주기는 합니다. 하지만 최종 분석에서는 결국 원본과 맞대 봐야 합니다. 그래서 respondent_id, evidence, needs_review 같은 열을 늘 챙겨서 요구하는 습관이 중요합니다. 응답자 ID는 원본 행을 찾는 주소이고, 근거 문구는 AI 판단을 확인하는 표시라고 보면 됩니다."

완료 판단 체크포인트:

수강생이 같은 데이터로 프롬프트 A/B/C를 비교하고, respondent_id와 evidence가 포함된 주제·감성 분류표를 만든 뒤 최종 프롬프트 선택 이유를 말하면 이번 차시를 완료한 것으로 본다.

예상 질문과 답변

Q1. AI가 분류한 주제는 정답인가요?

A. 정답이라기보다 초안입니다. 주관식 텍스트는 해석이 들어가기 때문에 코드북을 먼저 확인해야 합니다. 그다음 표본을 사람이 검토하고, 기준이 괜찮다고 판단되면 전체에 적용하는 절차가 필요합니다.

Q2. 감성은 NPS만 보고 정하면 안 되나요?

A. NPS는 중요한 참고값이지만 주관식 문장과 다를 수 있습니다. 낮은 NPS인데 긍정 문장을 쓴 사람도 있고, 높은 NPS인데 개선 요청을 쓴 사람도 있습니다. 그래서 감성은 점수만 보지 말고 문장 근거와 함께 봐야 합니다.

Q3. 주제는 몇 개가 적당한가요?

A. 이번 수업에서는 6-8개가 적당합니다. 너무 적으면 여러 의견이 한 덩어리로 뭉뚱그려지고, 너무 많으면 나중에 집계하기 어렵습니다. 실무에서는 표본을 보며 주제 수를 조정합니다.

Q4. 파일 업로드가 안 되면 어떻게 하나요?

A. 핸드아웃의 표본 18행을 복사해 실습합니다. 이때는 반드시 "표본 18행 기준"이라고 쓰게 해야 합니다. 표본 결과를 전체 1,500행의 결론으로 말하면 안 됩니다.

Q5. AI가 원문에 없는 원인을 만들어 냅니다.

A. 프롬프트에 "원문에 없는 원인, 의도, 고객 특성은 추측하지 말라"를 넣어 보세요. 또 evidence 열을 요구해야 합니다. 근거 문구가 있으면 원문에 없는 판단을 더 쉽게 찾아낼 수 있습니다.

Q6. 같은 프롬프트를 다시 실행하면 조금 다른 결과가 나옵니다.

A. 그럴 수 있습니다. 그래서 코드북을 먼저 고정하고, 출력 컬럼과 기준을 명확히 해야 합니다. 중요한 업무라면 AI 결과를 그대로 쓰지 말고 사람이 표본을 검수해야 합니다.

Q7. 오류가 섞인 보조 설문 파일은 왜 오늘 본 실습에 쓰지 않나요?

A. 오늘 목표는 프롬프트 구조를 배우는 것입니다. 오류가 섞인 보조 설문 파일에는 결측, 중복, 문자 혼입, 표기 혼재가 있어 별도의 정제 수업이 필요합니다. 다만 데이터 상태를 프롬프트에 알려야 하는 이유를 설명할 때 보조 사례로 사용합니다.

강의 운영 메모

  • 수강생이 AI 답변 문장만 보고 만족하면 respondent_id와 evidence가 있는지 되묻는다.
  • 표본 실습자는 전체 결론처럼 쓰지 않도록 계속 확인한다.
  • "좋은 프롬프트"를 길이로 판단하지 않게 한다. 길어도 기준이 흐리면 좋은 프롬프트가 아니다.
  • 프롬프트 C는 한 번에 긴 요청을 쓰는 것이 아니라 단계적으로 대화를 이어 가는 방식임을 강조한다.
  • 개인정보가 있는 실제 설문은 업로드 전에 반드시 제거해야 한다고 안내한다.

S08. 모범답안

미션 해설

미션은 같은 설문 데이터 또는 같은 표본 18행을 놓고 프롬프트 A/B/C를 비교하는 일입니다. 여기서 정답 문장이 딱 하나로 정해지는 것은 아닙니다. 대신 좋은 답안인지 보려면, AI에게 일을 맡기기 전에 기준을 얼마나 분명하게 알려 주었는지를 봐야 합니다.

좋은 답안에는 아래 내용이 들어가야 합니다.

  1. 데이터 맥락을 설명한다.
  2. AI가 해야 할 질문을 좁힌다.
  3. 결과 표의 컬럼을 지정한다.
  4. 원문 근거와 응답자 ID를 요구한다.
  5. 검토 필요 행을 찾게 한다.
  6. 표본 분석과 전체 분석을 구분한다.

왜 이런 조건이 필요할까요? AI에게 그냥 “분석해줘”라고 하면 AI는 그럴듯한 요약을 만들 수는 있습니다. 하지만 우리가 원하는 기준으로 분류했는지, 나중에 원본과 대조할 수 있는지, 엑셀에 붙여 넣어 다시 집계할 수 있는지는 보장되지 않습니다. 그래서 좋은 프롬프트는 질문만 던지는 문장이 아니라, 분석 요청서에 가깝습니다.

예시 답안: 프롬프트 A

이 설문 데이터를 분석해줘.

평가:

프롬프트 A는 일부러 부족하게 만든 프롬프트입니다. 데이터가 무엇인지, 어떤 질문에 답해야 하는지, 결과를 어떤 모양으로 받아야 하는지가 모두 빠져 있습니다.

이렇게 물어도 AI가 답을 안 하는 것은 아닙니다. 다만 문제는 답이 너무 자유롭게 나온다는 데 있습니다. 예를 들어 주제를 어떤 기준으로 나눴는지 확인하기 어렵고, 결과를 엑셀에 붙여 넣어 피벗테이블로 다시 집계하기도 어렵습니다. 초보자 입장에서는 그럴듯해 보이는 문장에 속기 쉬우니, 이런 프롬프트는 연습용으로만 보고 실제 분석에는 그대로 쓰지 않는 편이 좋습니다.

예시 답안: 프롬프트 B

너는 고객경험 분석가다.
데이터는 온라인 쇼핑 서비스의 고객 설문이다. 한 행은 응답 1건이다.
주요 컬럼은 respondent_id, age_group, channel, nps, satisfaction, free_text다.

목표는 free_text를 읽고 고객 의견을 운영 개선에 쓸 수 있게 정리하는 것이다.
각 응답을 주제와 감성으로 분류해줘.

주제는 6-8개로 만들되 너무 세분화하지 마라.
감성은 긍정, 부정, 혼합, 중립 중 하나로 정하라.
NPS와 free_text의 분위기가 어긋나면 혼합 또는 검토 필요로 표시하라.

출력은 표로 작성하라.
표 컬럼은 respondent_id, topic, sentiment, evidence, confidence, needs_review로 해라.
evidence에는 판단 근거가 된 원문 일부를 넣어라.
confidence는 높음, 중간, 낮음 중 하나로 표시하라.

평가:

프롬프트 B는 수업 기준으로 통과 가능한 프롬프트입니다. 온라인 쇼핑 서비스의 고객 설문이라는 맥락이 있고, free_text를 주제와 감성으로 분류하라는 질문도 들어 있습니다. 또 결과를 표로 받으라고 했고, respondent_id와 evidence를 요구했기 때문에 나중에 원본과 대조할 수 있습니다.

다만 아직 아쉬운 점이 있습니다. 주제를 나누는 기준, 즉 코드북을 먼저 확인하지 않고 바로 행별 분류로 들어갑니다. 코드북은 분류할 때 쓰는 사전 같은 것입니다. 예를 들어 “결제/혜택”과 “앱/사용성”을 어떻게 구분할지 미리 정해 두는 기준표라고 보면 됩니다. 이 기준표를 먼저 확인하지 않으면, AI가 만든 주제 기준이 마음에 들지 않을 때 전체 결과를 다시 받아야 합니다.

예시 답안: 프롬프트 C

아래 설문 데이터를 분석하되, 한 번에 결론을 내지 말고 3단계로 진행해줘.

데이터 맥락:
- 온라인 쇼핑 서비스의 고객 설문이다.
- 한 행은 응답 1건이다.
- 주요 컬럼은 respondent_id, age_group, channel, nps, satisfaction, free_text다.
- 목표는 주관식 의견을 운영 개선에 쓸 수 있게 분류하는 것이다.

1단계:
free_text를 분류할 주제 코드북을 먼저 제안해줘.
주제는 6-8개로 제한하고, 각 주제의 정의와 포함 예시를 적어줘.
서로 겹치기 쉬운 주제는 구분 기준도 적어줘.
이 단계에서는 행별 분류를 하지 마라.

2단계:
내가 코드북을 확인한 뒤, 그 코드북으로 각 행을 분류해줘.
출력 표 컬럼은 respondent_id, topic, sentiment, evidence, confidence, needs_review로 해라.
sentiment는 긍정, 부정, 혼합, 중립 중 하나로 정하라.
NPS와 free_text가 어긋나면 혼합 또는 needs_review=예로 표시하라.
원문에 없는 원인이나 고객 의도는 추측하지 마라.

3단계:
분류 결과를 주제별 응답수, 비중, 평균 NPS, 평균 satisfaction, 대표 근거문구로 요약해줘.
마지막에는 사람이 다시 확인해야 할 행 5개를 respondent_id와 이유로 정리해줘.

파일이 아니라 표본 데이터만 사용한 경우에는 모든 결론 앞에 "표본 기준"이라고 표시해줘.

평가:

프롬프트 C가 가장 안정적입니다. 한 번에 결론부터 내지 않고, 기준을 만들고, 그 기준으로 분류하고, 마지막에 요약하도록 순서를 나누었기 때문입니다.

여기서 중요한 장치는 코드북입니다. 코드북을 먼저 만들게 하면 AI가 어떤 기준으로 주제를 나누려는지 사람이 먼저 확인할 수 있습니다. 마치 시험 채점 전에 채점 기준표를 먼저 보는 것과 비슷합니다. 기준이 괜찮은지 확인한 뒤에 행별 분류로 넘어가면 결과를 훨씬 더 믿고 검토할 수 있습니다.

또한 respondent_id, evidence, needs_review가 들어 있어 원본 대조와 사람 검토가 가능합니다. respondent_id는 사전에서 단어를 다시 찾을 때 쓰는 표제어처럼, 분석 결과를 원래 응답 행으로 되돌려 확인하게 해 주는 열입니다.

A/B/C 비교표 예시

항목 프롬프트 A 프롬프트 B 프롬프트 C
맥락이 충분한가 부족 있음 충분
질문이 구체적인가 부족 구체적 단계별로 구체적
출력형식이 표로 고정되는가 불안정 고정 고정
원문 근거가 포함되는가 보장 안 됨 포함 요청 포함 요청
검토 필요 행을 찾는가 보장 안 됨 일부 가능 명시적으로 요청
업무에 바로 쓰기 좋은가 낮음 보통 이상 가장 좋음

정리 문장 예시:

내가 선택한 최종 프롬프트는 C이며, 이유는 코드북을 먼저 확인한 뒤 같은 기준으로 행별 분류와 요약을 진행할 수 있기 때문이다. 특히 respondent_id와 evidence 열이 있어 원본과 대조할 수 있다.

이 문장은 왜 C를 골랐는지 분명히 말하고 있습니다. 단순히 “C가 좋아 보인다”가 아니라, 코드북 확인, 같은 기준 적용, 원본 대조라는 이유를 들어 설명합니다. 모범답안에서는 이렇게 선택 이유가 구체적으로 드러나야 합니다.

검증용 코드 실행 결과 요약

아래 코드는 AI 답변을 대신하는 정답 코드가 아닙니다. AI가 만든 분류 결과를 그대로 믿기 전에, 데이터 규모와 표본이 맞는지, 간단한 규칙으로 보면 어떤 흐름이 나오는지 확인하기 위한 검사용 코드입니다.

실습 코드

아래 파이썬 코드는 검증용으로 실행한 코드입니다. 수강생이 꼭 이 코드를 외울 필요는 없습니다. 다만 코드가 하는 일은 알아두면 좋습니다. 데이터를 읽고, 행과 열 수를 확인하고, NPS 그룹을 나누고, 간단한 키워드 규칙으로 주제와 감성을 붙인 뒤 요약표를 만드는 흐름입니다.

from pathlib import Path

import pandas as pd


ROOT = Path("/mnt/d/ai-data-analysis-course")
SURVEY_CLEAN = ROOT / "02_datasets/survey_responses/survey_responses_clean.csv"
SURVEY_DIRTY = ROOT / "02_datasets/survey_responses/survey_responses_dirty.csv"
MOVIES = ROOT / "02_datasets/real_korean/movie_box/dacon_movies_train.csv"

EXPECTED_SURVEY_COLUMNS = [
    "respondent_id",
    "age_group",
    "gender",
    "nps",
    "satisfaction",
    "channel",
    "free_text",
    "submitted_at",
]


def nps_group(score: int) -> str:
    if score <= 6:
        return "비추천(0-6)"
    if score <= 8:
        return "중립(7-8)"
    return "추천(9-10)"


def classify_topic(text: str) -> str:
    rules = [
        ("배송/픽업", ["배송", "포장", "픽업", "기사님"]),
        (
            "고객지원/응대",
            ["고객센터", "상담", "문의", "답변", "처리", "교환", "환불", "반품", "응대", "문제 발생"],
        ),
        ("결제/혜택", ["결제", "쿠폰", "할인", "포인트", "혜택", "회원", "가격"]),
        ("상품정보/품질", ["상품 설명", "상품 비교", "품질", "실제 구성", "상세 페이지", "사진"]),
        ("매장/오프라인", ["매장", "오프라인", "재고", "헛걸음"]),
        ("알림/마케팅", ["알림", "푸시", "이벤트", "프로모션", "추천 상품"]),
        (
            "앱/사용성",
            ["모바일", "앱", "화면", "검색", "필터", "로딩", "메뉴", "디자인", "주소", "자동완성", "가입", "이전 구매", "예약", "정보", "글자"],
        ),
    ]
    for topic, keywords in rules:
        if any(keyword in text for keyword in keywords):
            return topic
    return "기타"


def classify_text_sentiment(text: str) -> str:
    negative_or_improve = [
        "불편",
        "헷갈",
        "오류",
        "늦",
        "오래",
        "길",
        "복잡",
        "부담",
        "개선",
        "재시도",
        "어렵",
        "달라",
        "많아서",
        "더 좋",
        "있으면",
        "아쉬",
        "표시되면",
        "바꿨습니다",
    ]
    positive = [
        "좋",
        "편리",
        "정확",
        "만족",
        "쉬",
        "친절",
        "명확",
        "빨",
        "안심",
        "신뢰",
        "깔끔",
        "직관",
        "잘",
        "도움",
        "부담이 없",
        "해결",
        "일치",
    ]
    has_negative = any(word in text for word in negative_or_improve)
    has_positive = any(word in text for word in positive)

    if has_negative and has_positive:
        return "혼합/개선"
    if has_negative:
        return "부정/개선"
    if has_positive:
        return "긍정"
    return "중립"


def markdown_table(df: pd.DataFrame) -> str:
    cols = list(df.columns)
    lines = [
        "| " + " | ".join(cols) + " |",
        "| " + " | ".join(["---"] * len(cols)) + " |",
    ]
    for _, row in df.iterrows():
        lines.append("| " + " | ".join(str(row[col]).replace("|", "/") for col in cols) + " |")
    return "\n".join(lines)


def main() -> None:
    survey = pd.read_csv(SURVEY_CLEAN)
    dirty = pd.read_csv(SURVEY_DIRTY)
    movies = pd.read_csv(MOVIES)

    assert list(survey.columns) == EXPECTED_SURVEY_COLUMNS
    assert survey.shape == (1500, 8)
    assert dirty.shape == (1510, 8)
    assert movies.shape == (600, 12)
    assert survey["free_text"].isna().sum() == 0

    survey = survey.copy()
    survey["nps_group"] = survey["nps"].apply(nps_group)
    survey["topic_rule"] = survey["free_text"].apply(classify_topic)
    survey["text_sentiment_rule"] = survey["free_text"].apply(classify_text_sentiment)

    sample_parts = []
    for group in ["비추천(0-6)", "중립(7-8)", "추천(9-10)"]:
        part = survey.loc[survey["nps_group"] == group].sample(n=6, random_state=8)
        sample_parts.append(part)
    prompt_sample = pd.concat(sample_parts).sort_values(["nps", "respondent_id"])

    topic_summary = (
        survey.groupby("topic_rule")
        .agg(
            응답수=("respondent_id", "count"),
            평균NPS=("nps", "mean"),
            평균만족도=("satisfaction", "mean"),
        )
        .round({"평균NPS": 2, "평균만족도": 2})
        .sort_values(["응답수", "평균NPS"], ascending=[False, True])
        .reset_index()
        .rename(columns={"topic_rule": "주제"})
    )

    sentiment_summary = (
        survey["text_sentiment_rule"]
        .value_counts()
        .rename_axis("주관식_감성_휴리스틱")
        .reset_index(name="응답수")
    )

    dirty_issues = {
        "dirty_rows": len(dirty),
        "duplicated_respondent_id": int(dirty.duplicated("respondent_id").sum()),
        "missing_nps": int(dirty["nps"].isna().sum()),
        "missing_free_text": int(dirty["free_text"].isna().sum()),
        "non_numeric_satisfaction": int(pd.to_numeric(dirty["satisfaction"], errors="coerce").isna().sum()),
        "age_group_variants_for_20s": int(dirty["age_group"].isin(["20s", "이십대"]).sum()),
    }

    genre_summary = (
        movies.groupby("genre")
        .agg(영화수=("title", "count"), 평균관객수=("box_off_num", "mean"))
        .round({"평균관객수": 0})
        .astype({"평균관객수": "int64"})
        .sort_values("영화수", ascending=False)
        .head(5)
        .reset_index()
    )

    print("pandas version:", pd.__version__)
    print("survey clean shape:", survey.shape[0], "rows x", 8, "columns")
    print("survey dirty shape:", dirty.shape[0], "rows x", dirty.shape[1], "columns")
    print("dacon movies train shape:", movies.shape[0], "rows x", movies.shape[1], "columns")

    print("\n[dirty issue counts]")
    for key, value in dirty_issues.items():
        print(f"{key}: {value}")

    print("\n[NPS group counts]")
    print(survey["nps_group"].value_counts().reindex(["비추천(0-6)", "중립(7-8)", "추천(9-10)"]).to_string())

    print("\n[topic summary by simple rules]")
    print(topic_summary.to_string(index=False))

    print("\n[text sentiment summary by simple rules]")
    print(sentiment_summary.to_string(index=False))

    print("\n[prompt sample 18 rows]")
    sample_cols = ["respondent_id", "age_group", "channel", "nps", "satisfaction", "free_text"]
    print(markdown_table(prompt_sample[sample_cols]))

    print("\n[optional movie genre top 5]")
    print(genre_summary.to_string(index=False))


if __name__ == "__main__":
    main()

이 코드에서 눈여겨볼 부분은 세 가지입니다. nps_group은 점수를 비추천, 중립, 추천으로 나눕니다. classify_topic은 주관식 문장 안의 단어를 보고 임시 주제를 붙입니다. classify_text_sentiment는 긍정 표현과 부정 또는 개선 표현을 찾아 감성을 나눕니다.

하지만 이 방식은 어디까지나 단순한 기준선입니다. 단어 하나만 보고 판단하기 때문에 문맥을 깊게 읽지는 못합니다. 그래서 실제 수업에서는 이 코드 결과를 정답으로 외우기보다, AI 프롬프트를 어떻게 설계해야 검토 가능한 결과를 받을 수 있는지에 더 집중해 보세요.
